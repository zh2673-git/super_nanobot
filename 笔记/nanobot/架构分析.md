# nanobot 架构分析

## 第一步：点出核心组件概念

nanobot 的架构由 **5 个核心组件** 构成，它们就像一个精密协作的团队：

- **AgentLoop（代理循环）**：整个系统的"大脑"，负责接收消息、思考、调用工具、返回响应——就像团队的项目经理，统筹全局
- **MessageBus（消息总线）**：连接各渠道与 Agent 的"神经系统"——就像团队的通信员，确保信息准确传递
- **ToolRegistry（工具注册表）**：管理所有可用工具——就像团队的技术专家，提供各种执行能力
- **ProviderRegistry（提供商注册表）**：管理 LLM 提供商——就像团队的顾问团，提供智能支持
- **ChannelManager（渠道管理器）**：管理所有聊天渠道——就像团队的外联部门，负责与外界沟通

**联系说明**：用户通过聊天渠道发送消息，ChannelManager（外联部门）接收并转换为标准格式，通过 MessageBus（通信员）传递给 AgentLoop（项目经理）。项目经理根据上下文和记忆，咨询 ProviderRegistry（顾问团）获取智能回复，并通过 ToolRegistry（技术专家）执行具体操作。MCP 服务器提供了扩展外部工具的能力。

## 核心特性

- **Ultra-Lightweight**: 仅 ~4,000 行代码
- **Research-Ready**: 代码清晰易读，易于研究和扩展
- **Lightning Fast**: 最小占用，快速启动
- **Easy-to-Use**: 一键部署
- **MCP Support**: 支持 Model Context Protocol
- **Thinking Mode**: 支持 LLM 思考模式 (reasoning_effort)
- **v0.1.4.post4 更新**:
  - 更安全的默认访问控制
  - 更好的多实例支持
  - 更健壮的 MCP 支持
  - 渠道和提供商稳定性改进

## 消息流

### 消息处理流程

```
1. 用户通过渠道发送消息
   ↓
2. Channel 接收消息，转换为 InboundMessage
   ↓
3. MessageBus (Async Queue) 路由消息
   ↓
4. AgentLoop 获取消息
   ↓
5. ContextBuilder 组装上下文
   ↓
6. LiteLLMProvider 调用 LLM
   ↓
7. ToolRegistry 执行工具调用
   ↓
8. 返回结果给 LLM
   ↓
9. 生成响应，转换为 OutboundMessage
   ↓
10. ChannelManager 路由到目标渠道
    ↓
11. 渠道发送响应给用户
```
