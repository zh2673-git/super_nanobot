# OllamaPilot 运行逻辑

> 从用户输入到最终输出的完整流程解析

## 🎯 第一步：认识执行流程的五个阶段

当你在命令行输入"明天苏州天气怎么样？"时，OllamaPilot 会经历五个处理阶段，就像一条精密的流水线：

- **阶段一：用户输入接收**：接收并预处理用户的查询，像快递收件台接收包裹（清理输入、识别意图、准备初始状态）
- **阶段二：Skill 智能选择**：根据关键词选择合适的 Skill，像分拣中心根据地址选择配送路线（匹配触发词、激活对应 Skill、注入系统提示词）
- **阶段三：模型推理与工具调用**：模型决定调用哪些工具，像配送员根据路线使用不同的交通工具（搜索、抓取网页、执行代码等）
- **阶段四：中间件处理**：中间件链处理请求和结果，像沿途的检查站（日志记录、失败重试、限制调用次数）
- **阶段五：结果整合与返回**：整理工具结果并返回给用户，像快递送达后的签收确认（格式化结果、保存记忆、清理状态）

**为什么需要这些阶段？**

想象一个没有流程的系统：
- 用户问"天气"，模型不知道该干什么
- 工具调用失败了，没有重试机制
- 没有日志，出了问题无法调试
- 对话没有记忆，每次都从零开始

OllamaPilot 的五阶段流程确保了：
- 每个问题都能找到合适的处理器
- 工具调用稳定可靠
- 执行过程可追踪
- 对话能够连续

## 📖 第三步：深入理解每个阶段

### 阶段一：用户输入接收

**专业定义**：接收用户输入，创建 LangChain 消息对象，初始化对话状态，为后续处理做准备。

**通俗解释与类比**：
这就像快递收件台接收包裹的过程：

1. **接收包裹**：用户输入"明天苏州天气怎么样？"
2. **填写面单**：创建 `HumanMessage` 对象，记录包裹信息
3. **初始扫描**：清理输入（去除多余空格、特殊字符）
4. **准备派送**：创建初始状态 `{"messages": [HumanMessage]}`

**在代码中的体现**：
```python
# agent.py：agent.invoke() 方法
def invoke(self, query: str, thread_id: Optional[str] = None) -> str:
    if self.verbose:
        print(f"🤖 用户: {query}")
    
    # 配置对话 ID
    config = {"configurable": {"thread_id": thread_id or "default"}}
    
    # 创建消息并执行
    result = self.agent.invoke(
        {"messages": [HumanMessage(content=query)]},
        config
    )
    
    # 提取最终回复
    messages = result.get("messages", [])
    last_message = messages[-1]
    return last_message.content
```

**关键点**：
- **HumanMessage**：代表用户消息，是 LangChain 的标准消息类型
- **thread_id**：对话线程 ID，用于区分不同的对话会话
- **初始化状态**：`{"messages": [...]}` 是 Agent 的标准状态格式

### 阶段二：Skill 智能选择

**专业定义**：通过 SkillSelectorMiddleware 在 before_model 阶段，根据用户查询匹配触发词，选择合适的 Skill，并注入对应的系统提示词。

**通俗解释与类比**：
这就像快递分拣中心根据地址选择配送路线：

1. **扫描地址**：提取用户查询中的关键词
2. **匹配路线**：遍历所有配送路线（Skill），找到匹配的
3. **指定配送员**：激活对应的 Skill
4. **发放配送说明**：注入系统提示词，告诉模型如何处理

**匹配算法详解**：

```python
# skill_middleware.py：_select_skill 方法
def _select_skill(self, query: str) -> Optional[Skill]:
    """
    匹配流程：
    1. 将查询转为小写："明天苏州天气怎么样" → "明天苏州天气怎么样"
    2. 遍历所有 Skill 的触发词：
       - Weather: ["天气", "温度"] → "天气" in "明天苏州天气怎么样" ✓
       - Python: ["python", "代码"] → 不匹配 ✗
    3. 返回第一个匹配的 Skill：Weather Skill
    """
    # 1. 查找匹配的特定 Skill
    matches = self.registry.find_skill_by_trigger(query)
    
    if matches:
        return self.registry.get_skill(matches[0])
    
    # 2. 返回默认 Skill
    return self.registry.get_skill(self.default_skill_name)
```

**系统提示词注入**：

```python
# skill_middleware.py：before_model 方法
def before_model(self, state, runtime):
    # 选择 Skill
    skill = self._select_skill(query)
    
    # 获取系统提示词
    system_prompt = skill.get_system_prompt()
    
    # 检查是否已存在系统消息
    has_system = any(isinstance(msg, SystemMessage) for msg in messages)
    
    if not has_system:
        # 添加系统消息到消息列表开头
        return {
            "messages": [SystemMessage(content=system_prompt)] + messages
        }
```

**SystemMessage 的作用**：
- 像"工作任务书"，告诉模型应该干什么
- 定义了模型的角色、能力、限制
- Weather Skill 的系统提示词示例：
  ```yaml
  你是天气查询专家，帮助用户获取天气信息。
  
  工作流程：
  1. 使用 web_search 搜索城市天气
  2. 使用 web_fetch 获取详细页面
  3. 整理天气信息返回给用户
  ```

### 阶段三：模型推理与工具调用

**专业定义**：模型根据系统提示词和对话历史，决定是否调用工具、调用哪个工具、传递什么参数，并生成工具调用请求。

**通俗解释与类比**：
这就像配送员根据任务书决定使用什么交通工具：

1. **查看任务书**：模型读取系统提示词
2. **理解任务**：模型理解要查询天气
3. **选择工具**：模型决定使用 `web_search` 工具
4. **准备参数**：模型生成参数 `{"query": "苏州明天天气"}`
5. **执行工具**：系统调用工具并获取结果
6. **继续推理**：模型根据结果决定下一步

**工具调用的循环机制**：

```
┌─────────────────┐
│  模型推理        │
│  ↓               │
│  决策：调用工具？ │
└─────────────────┘
         ↓ 是
┌─────────────────┐
│  生成工具调用    │
│  ToolCall(...)  │
└─────────────────┘
         ↓
┌─────────────────┐
│  执行工具        │
│  web_search()   │
└─────────────────┘
         ↓
┌─────────────────┐
│  返回结果        │
│  ToolMessage()  │
└─────────────────┘
         ↓
    回到模型推理 ─┘
         ↓ 否（信息足够）
┌─────────────────┐
│  生成最终回复    │
│  AIMessage()    │
└─────────────────┘
```

**关键代码流程**：
```python
# LangChain 内部的工具调用循环（简化版）
while True:
    # 1. 模型推理
    response = model.invoke(messages)
    
    # 2. 检查是否需要调用工具
    if response.tool_calls:
        for tool_call in response.tool_calls:
            # 3. 执行工具
            result = execute_tool(tool_call)
            
            # 4. 添加工具结果到消息历史
            messages.append(ToolMessage(content=result))
    else:
        # 5. 返回最终回复
        return response
```

### 阶段四：中间件处理

**专业定义**：中间件在 Agent 执行的特定阶段插入自定义逻辑，处理工具调用、记录日志、实现重试、限制调用次数等。

**通俗解释与类比**：
这就像沿途的检查站：

1. **记录站**（ToolLoggingMiddleware）：记录每个工具的调用和结果
2. **重试站**（ToolRetryMiddleware）：工具失败时重新派单
3. **限流站**（ToolCallLimitMiddleware）：防止无限循环

**中间件的执行时机**：

```python
# Agent 执行流程中的中间件调用点
def agent_loop():
    # ===== before_model 阶段 =====
    for middleware in middlewares:
        if hasattr(middleware, 'before_model'):
            middleware.before_model(state, runtime)
    
    # ===== 模型推理 =====
    response = model.invoke(messages)
    
    # ===== 处理工具调用 =====
    for tool_call in response.tool_calls:
        # ===== wrap_tool_call 阶段 =====
        for middleware in middlewares:
            if hasattr(middleware, 'wrap_tool_call'):
                result = middleware.wrap_tool_call(tool_call, handler)
            else:
                result = handler(tool_call)
```

**ToolRetryMiddleware 的工作原理**：

```python
# 伪代码：重试中间件的实现
def wrap_tool_call(self, request, handler):
    max_retries = 2
    for attempt in range(max_retries + 1):
        try:
            result = handler(request)
            return result
        except Exception as e:
            if attempt < max_retries:
                # 重试
                continue
            else:
                # 重试次数用尽，抛出异常
                raise e
```

**ToolCallLimitMiddleware 的工作原理**：

```python
# 伪代码：调用限制中间件的实现
def __init__(self, run_limit=50):
    self.run_limit = run_limit
    self.call_count = 0

def before_model(self, state, runtime):
    self.call_count += 1
    if self.call_count > self.run_limit:
        raise Exception(f"工具调用次数超过限制 {self.run_limit}")
```

### 阶段五：结果整合与返回

**专业定义**：提取模型的最终回复，保存对话记忆，清理状态，返回结果给用户。

**通俗解释与类比**：
这就像快递送达后的签收确认：

1. **确认签收**：提取模型的最终回复
2. **记录存档**：保存对话记忆（用于下次对话）
3. **清理状态**：重置临时状态
4. **通知客户**：返回结果给用户

**对话记忆的保存**：

```python

```

## 🔧 第五步：关键机制详解

### 机制一：工具调用循环

**核心逻辑**：模型不断推理，直到信息足够生成最终回复。

```python
# 伪代码：工具调用循环
def run_agent_loop(messages):
    while True:
        # 1. 模型推理
        response = model.invoke(messages)
        
        # 2. 检查是否需要工具
        if not response.tool_calls:
            # 3. 不需要工具，返回最终回复
            return response.content
        
        # 4. 需要工具，执行循环
        for tool_call in response.tool_calls:
            # 执行工具
            result = execute_tool(tool_call)
            
            # 添加工具结果
            messages.append(ToolMessage(content=result))
        
        # 5. 继续下一轮推理
```

**防止无限循环**：
- ToolCallLimitMiddleware：限制最多 50 次工具调用
- 避免模型陷入重复调用工具的死循环

### 机制三：错误处理与重试

**ToolRetryMiddleware 的实现**：

```python
# skill_middleware.py：ToolLoggingMiddleware
def wrap_tool_call(self, request, handler):
    # 记录调用
    print(f"🔧 执行工具: {tool_name}({tool_args})")
    
    try:
        # 执行工具
        result = handler(request)
        
        # 记录成功
        print(f"   ✅ 结果: {result_preview}")
        return result
    except Exception as e:
        # ToolRetryMiddleware 会在这里重试
        print(f"   ❌ 失败: {e}")
        raise
```

**重试流程**：
```
第1次调用失败
    ↓
ToolRetryMiddleware 捕获异常
    ↓
自动重试（第2次调用）
    ↓
如果成功 → 继续执行
如果失败 → 再次重试（第3次调用）
    ↓
如果仍失败 → 返回错误信息
```

## 📊 第六步：性能优化建议

### 优化一：减少不必要的工具调用

**问题**：模型可能调用多次类似工具，浪费资源。

**解决**：优化系统提示词，明确工作流程。

```yaml
# SKILL.md 中明确的流程
工作流程：
  1. 使用 web_search 搜索城市天气（只搜索一次）
  2. 使用 web_fetch 获取详细页面（只获取一次）
  3. 整理天气信息返回给用户（不要重复搜索）
```

### 优化二：缓存常用结果

**问题**：相同查询重复调用工具。

**解决**：添加结果缓存机制（可扩展）。

```python
# 伪代码：工具结果缓存
cache = {}

def web_search_with_cache(query):
    if query in cache:
        return cache[query]
    
    result = web_search.invoke(query)
    cache[query] = result
    return result
```

### 优化三：监控工具调用次数

**问题**：频繁的工具调用可能导致性能下降。

**解决**：使用 verbose 模式监控调用次数。

```python
# agent.py：启用 verbose 模式
agent = create_agent(model, verbose=True)
# 会输出每个工具调用的详细信息
```
