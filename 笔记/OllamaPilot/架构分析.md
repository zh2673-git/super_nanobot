# OllamaPilot 架构分析

> 基于 LangChain 1.0+ 原生实现，模块化设计，小模型优化

---

## 🎯 第一步：认识架构的三大支柱

OllamaPilot 的架构设计就像建一栋现代化办公大楼，有三大支柱支撑整个系统：

- **LangChain 原生Agent架构**：基于 `create_agent` 工厂函数，像大楼的钢筋混凝土框架（提供稳固的基础结构，让整个系统有清晰的骨架）
- **AgentMiddleware 中间件机制**：可插拔的处理流程，像大楼的智能系统（照明、空调、电梯，各司其职，互不干扰）
- **USB 式 Skill 设计**：即插即用的功能模块，像办公室的工位（可以随时增减，不影响其他工位的工作）

**为什么这样设计？**

传统 AI Agent 开发的问题：
- 所有功能堆在一起，像一间大杂烩办公室
- 改一个功能可能影响其他功能，像牵一发而动全身
- 难以扩展，像老房子很难加装新设备

OllamaPilot 的解决方案：
- 使用 LangChain 标准架构，像用建筑标准规范
- 用中间件分离关注点，像用模块化设计
- 用 Skill 封装功能，像用独立工位

## 📖 第三步：深入理解每个架构层次

### 一、LangChain 原生Agent架构

**专业定义**：OllamaPilot 基于 LangChain 1.0+ 的 `create_agent` 工厂函数构建，使用 LangGraph 的 `MemorySaver` 实现对话记忆，完全遵循 LangChain 的设计哲学和最佳实践。

**通俗解释与类比**：
想象你要建一座房子：
- **传统方式**：自己从零开始设计框架、打地基、砌墙，每个细节都要自己操心
- **LangChain 方式**：使用建筑公司的标准化框架，就像买了一套"建筑积木套装"
  - 框架已经设计好（`create_agent` 提供了标准的创建流程）
  - 墙壁可以随意组合（Middleware 可以自由添加）
  - 家具可以即插即用（Skill 和 Tool 可以随时添加）

**在项目中的具体体现**：
`agent.py` 中的核心代码：

```python
# 使用 LangChain 原生 create_agent
self.agent = lc_create_agent(
    model=model.bind_tools(self.builtin_tools),  # 绑定工具到模型
    tools=self.builtin_tools,                     # 工具列表
    middleware=middleware,                        # 中间件列表
    checkpointer=self.checkpointer,               # 记忆持久化
)
```

**关键设计点**：
1. **工具绑定**：`model.bind_tools()` 让模型知道可以使用哪些工具
2. **中间件注入**：`middleware=[]` 注入自定义处理逻辑
3. **记忆持久化**：`checkpointer=MemorySaver()` 实现对话记忆

### 二、AgentMiddleware 中间件机制

**专业定义**：AgentMiddleware 是 LangChain 提供的中间件接口，允许在 Agent 执行的不同阶段（before_model、wrap_tool_call 等）注入自定义逻辑。OllamaPilot 用它实现 Skill 选择、工具日志、自动重试等功能。

**通俗解释与类比**：
想象一条河流：
- 河水流向大海（请求流向模型）
- 沿途可以建水坝、水电站、监测站（插入中间件）
- 每个设施独立工作，互不影响
- 可以根据需要增减设施数量和顺序

中间件就像这些"河道设施"：
- **before_model**：水流经过前的处理（选择 Skill、注入提示词）
- **wrap_tool_call**：水流中的处理（记录日志、重试失败的工具）
- **after_model**：流出后的处理（整理结果）

**在项目中的具体体现**：
`skill_middleware.py` 中实现了多个中间件：

1. **SkillSelectorMiddleware**（在 before_model 阶段）：
   ```python
   def before_model(self, state, runtime):
       # 根据用户查询选择合适的 Skill
       skill = self._select_skill(query)
       # 注入 Skill 的系统提示词
       return {"messages": [SystemMessage(content=system_prompt)] + messages}
   ```
   - 像水电站的"分流闸"，根据水流特点选择不同的河道

2. **ToolLoggingMiddleware**（在 wrap_tool_call 阶段）：
   ```python
   def wrap_tool_call(self, request, handler):
       # 记录工具调用前的状态
       print(f"🔧 执行工具: {tool_name}({tool_args})")
       # 执行工具
       result = handler(request)
       # 记录工具调用后的结果
       print(f"   ✅ 结果: {result_preview}")
       return result
   ```
   - 像河流监测站，记录水流的速度和流量

3. **ToolRetryMiddleware**（LangChain 内置）：
   - 像水坝的泄洪机制，水位过高（工具失败）时自动重新调度

**中间件执行顺序的重要性**：

```
请求 → Middleware 1 → Middleware 2 → Middleware 3 → 模型
         ↓               ↓               ↓
      选择Skill       记录日志        检查限制
```

在 `agent.py` 中的中间件配置顺序：
```python
middleware = [
    SkillSelectorMiddleware,  # 1. 最先选择 Skill
    ToolLoggingMiddleware,    # 2. 然后记录日志
    ToolRetryMiddleware,      # 3. 工具失败时重试
    ToolCallLimitMiddleware,  # 4. 最后限制调用次数
]
```

### 三、USB 式 Skill 设计

**专业定义**：Skill 是独立的功能模块，遵循"单一职责"原则，通过标准接口（Skill 基类）与系统集成。可以像 USB 设备一样即插即用，无需修改核心代码。

**通俗解释与类比**：
想象你有多个 USB 设备：
- 鼠标、键盘、摄像头、打印机
- 每个设备都有自己的功能
- 插上就能用，拔掉也不影响其他设备
- 电脑不需要知道设备内部的实现细节

Skill 就像 USB 设备：
- 每个 Skill 是独立的功能模块
- 通过标准接口（USB 接口）与系统连接
- 可以随时添加、移除、替换

**在项目中的具体体现**：
`skills/base.py` 定义了 Skill 的标准接口：

```python
class Skill(ABC):
    """Skill 抽象基类 - 就像 USB 标准接口"""
    
    # 元数据（设备的基本信息）
    name: str = ""              # 设备名称
    description: str = ""       # 设备描述
    triggers: List[str] = []    # 触发条件
    
    # 必须实现的方法（设备的基本功能）
    @abstractmethod
    def get_tools(self) -> List[BaseTool]:
        """返回设备提供的工具"""
        pass
    
    # 可选实现的方法（设备的扩展功能）
    def get_system_prompt(self) -> Optional[str]:
        """返回设备的使用说明"""
        return None
```

**Skill 的三种形态**：

1. **Markdown Skill（SKILL.md）**：
   - 像即插即用的"读卡器"
   - 配置文件定义功能
   - 无需写代码，适合简单功能
   ```yaml
   ---
   name: weather
   description: 查询天气
   triggers: [天气]
   tools: [web_search]
   ---
   系统提示词...
   ```

2. **Python Skill**：
   - 像可编程的"开发板"
   - 可以添加自定义逻辑
   - 适合中等复杂度的功能
   ```python
   class MySkill(Skill):
       def get_tools(self):
           return [custom_tool]
   ```

3. **MCP Integration**：
   - 像外接的"服务器设备"
   - 通过网络连接外部服务
   - 适合复杂的企业级集成

## 🔧 第四步：核心技术选型

### 为什么选择 LangChain 1.0+？

| 对比项 | 自研框架 | LangChain 1.0+ |
|--------|---------|---------------|
| 开发成本 | 高（要从零实现） | 低（使用现成框架） |
| 维护成本 | 高（自己维护） | 低（社区维护） |
| 功能完整性 | 取决于开发投入 | 内置丰富功能 |
| 社区支持 | 无 | 强大的社区生态 |
| 学习曲线 | 陡峭（要理解全部细节） | 平缓（有文档和示例） |
| 升级成本 | 高（全部自己改） | 低（跟随框架升级） |

**OllamaPilot 的选择**：
```python
# 使用 LangChain 原生功能
from langchain.agents import create_agent
from langchain.agents.middleware import ToolRetryMiddleware
from langgraph.checkpoint.memory import MemorySaver
```

### 为什么用 AgentMiddleware？

**传统方式的问题**：
```python
# 硬编码的处理逻辑
def handle_query(query):
    if "天气" in query:
        return weather_skill.handle(query)
    elif "python" in query:
        return python_skill.handle(query)
    else:
        return default_skill.handle(query)
    
    # 如果要添加日志、重试，需要修改这个函数
    # 如果要添加新 Skill，也要修改这个函数
```

**Middleware 方式的优势**：
```python
# 模块化、可插拔的处理逻辑
middleware = [
    SkillSelectorMiddleware(registry),  # 负责选择 Skill
    ToolLoggingMiddleware(),            # 负责记录日志
    ToolRetryMiddleware(),              # 负责重试
]
agent = create_agent(model, middleware=middleware)

# 添加新功能：只需添加新中间件
middleware.insert(1, RateLimitMiddleware())  # 限流
```

### 为什么用 Skill 系统？

**传统方式的痛点**：
- 所有功能写在一个文件里，代码耦合严重
- 添加新功能要改核心代码
- 不同功能之间容易相互影响

**Skill 系统的优势**：
- 每个 Skill 是独立模块，职责单一
- 添加新 Skill 只需创建新文件
- Skill 之间互不干扰，易于测试和维护

## 📐 第五步：关键设计决策

### 决策一：Skill 如何被激活？

**问题**：当用户说"明天苏州天气怎么样"时，系统如何知道应该激活 Weather Skill？

**解决方案**：触发词匹配 + 默认回退

```
用户输入
    ↓
[分词处理]
    ↓
[关键词提取]："明天"、"苏州"、"天气"
    ↓
[遍历所有 Skill 的触发词]
    ↓
Weather Skill.triggers = ["天气", "温度", "下雨"]
    ↓
匹配成功："天气" ∈ 用户输入
    ↓
激活 Weather Skill
```

**如果没有匹配的 Skill**：
```python
# 返回默认 Skill
default_skill = registry.get_default_skill()
```

### 决策二：工具调用如何容错？

**问题**：工具调用可能失败（网络错误、参数错误等），如何保证系统稳定？

**解决方案**：多层容错机制

```
工具调用失败
    ↓
[第一层：ToolRetryMiddleware]
    ↓
自动重试（最多 2 次）
    ↓
如果仍然失败
    ↓
[第二层：错误提示]
    ↓
返回友好的错误信息
    ↓
用户可以重新提问

防护措施：
[ToolCallLimitMiddleware]
    ↓
防止无限重试（最多 50 次工具调用）
```

### 决策三：如何管理对话记忆？

**问题**：Agent 需要记住之前的对话内容，如何持久化记忆？

**解决方案**：使用 LangGraph 的 MemorySaver

```python
# 对话记忆机制
checkpointer = MemorySaver()

# 使用 thread_id 区分不同对话
agent.invoke("我叫张三", thread_id="session_1")
agent.invoke("我叫什么名字？", thread_id="session_1")
# AI: 你叫张三

# 不同 thread_id 隔离对话
agent.invoke("你好", thread_id="session_2")
# AI: 你好！（不知道张三的事）
```

**生活类比**：
- `thread_id` 就像"会话ID"
- 不同的会话就像不同的聊天窗口
- 每个窗口记录独立的对话历史

### 决策四：如何优化小模型性能？

**问题**：4B 参数的小模型容易调用工具失败、陷入循环，如何优化？

**解决方案**：针对性优化

1. **模型参数优化**：
   ```python
   # models.py 中的小模型优化配置
   ChatOllama(
       num_ctx=8192,      # 足够的上下文窗口
       num_predict=2048,  # 限制生成长度
       top_p=0.9,         # 核采样
       top_k=40,          # TopK 采样
       repeat_penalty=1.1, # 防止重复
   )
   ```

2. **工具调用限制**：
   ```python
   # 防止无限循环
   ToolCallLimitMiddleware(run_limit=50)
   ```

3. **自动重试机制**：
   ```python
   # 自动纠正错误
   ToolRetryMiddleware(max_retries=2)
   ```

4. **清晰的系统提示词**：
   ```yaml
   # SKILL.md 中明确的指令
   你是天气查询专家，帮助用户获取天气信息。
   
   工作流程：
   1. 使用 web_search 搜索城市天气
   2. 使用 web_fetch 获取详细页面
   3. 整理天气信息返回给用户
   ```

## 🎓 架构设计总结

OllamaPilot 的架构设计遵循以下原则：

1. **标准化**：基于 LangChain 标准框架，易于理解和维护
2. **模块化**：使用 Middleware 和 Skill，功能独立、可插拔
3. **可扩展**：添加新功能无需修改核心代码
4. **容错性**：多层容错机制，系统稳定可靠
5. **小模型友好**：针对性优化，4B 模型也能稳定运行

**架构优势**：
- 代码简洁（核心代码少于 1000 行）
- 功能完整（工具丰富、记忆完善）
- 易于扩展（添加 Skill 和 Middleware 即可）
- 社区友好（遵循 LangChain 标准）
