# Open Deep Research 核心算法分析

## 核心概念

- **[Supervisor-Agent](Supervisor-Agent.md)**：Supervisor-Agent 模式，负责任务规划和执行的算法
- **[迭代式研究](迭代式研究.md)**：循环执行搜索和整合的算法
- **[结果压缩](结果压缩.md)**：整合多个研究单元结果的算法
- **[结构化输出](结构化输出.md)**：使用 Pydantic 进行结构化输出的算法

## 算法一：Supervisor-Agent 任务分配算法

### 算法概述

Supervisor-Agent 模式是整个系统的核心算法，它模拟了一个"项目经理-研究员"的工作模式。Supervisor（监督者）负责任务规划和进度追踪，Researcher（研究者）负责执行具体的研究任务。

**通俗解释**：这就像一个建筑项目，Supervisor 是项目经理，Researcher 是建筑工人。项目经理（Supervisor）决定需要做什么工作，然后分配给建筑工人（Researcher）；建筑工人完成后，项目经理检查进度，决定是否需要更多工作。

### 算法原理

```
输入：当前研究状态（research_iterations, research_findings）
输出：下一步操作（继续研究 / 结束研究）

1. 检查是否达到最大迭代次数
   如果 research_iterations >= max_researcher_iterations:
       返回 "compress_research"（结束研究）

2. 分析当前研究发现
   如果 research_findings 足够完整:
       返回 "compress_research"（结束研究）
   否则:
       返回 "conduct_research"（继续研究）

3. 分配研究任务
   将研究目标分解为多个子任务
   每个子任务分配给一个独立的研究单元
   并发执行所有研究单元
```

### 算法实现

关键代码位于 `research_supervisor` 函数中：

```python
async def research_supervisor(state: SupervisorState, config: RunnableConfig):
    # 第一步：检查迭代次数
    if state["research_iterations"] >= configurable.max_researcher_iterations:
        return Command(goto="compress_research")
    
    # 第二步：分析研究进度
    current_findings = state.get("research_findings", {})
    
    # 使用模型分析是否需要继续
    supervisor_model = configurable_model.with_config(model_config)
    supervisor_prompt = lead_researcher_prompt.format(
        research_brief=state["research_brief"],
        research_findings=current_findings,
        research_plan=state.get("research_plan", ""),
        max_iterations=configurable.max_researcher_iterations,
        current_iteration=state["research_iterations"],
    )
    
    supervisor_response = await supervisor_model.ainvoke([
        HumanMessage(content=supervisor_prompt)
    ])
    
    # 第三步：根据分析结果决定下一步
    if supervisor_response.should_continue:
        return Command(goto="conduct_research")
    else:
        return Command(goto="compress_research")
```

### 性能分析

- **时间复杂度**：O(n)，n 为迭代次数
- **空间复杂度**：O(k)，k 为并发研究单元数
- **优势**：支持并行执行，提高效率
- **局限**：受限于模型的理解能力和最大迭代次数

## 算法二：迭代式研究算法

### 算法概述

每个研究单元内部使用迭代式研究算法，循环执行搜索-评估-整合的过程，直到达到研究目标或迭代上限。

**通俗解释**：这就像剥洋葱一样，一层一层地深入研究。每搜索一次，就评估结果是否足够，如果不夠，就再搜索更深层次的信息。

### 算法原理

```
输入：研究任务（research_topic）
输出：研究发现（findings）

1. 初始化搜索查询队列
2. 当搜索队列不为空且未达到迭代上限:
   a. 从队列中获取下一个搜索查询
   b. 调用搜索 API 获取结果
   c. 处理和提取有用信息
   d. 评估是否需要更多搜索
   e. 如果需要，生成新的搜索查询并加入队列
3. 整合所有研究发现
4. 返回研究发现
```

### 算法实现

关键代码位于 `conduct_research` 函数中：

```python
async def conduct_research(state: ResearcherState, config: RunnableConfig):
    search_iteration = 0
    
    # 迭代搜索
    while search_iteration < configurable.max_search_iterations:
        # 1. 生成搜索查询
        query_prompt = generate_search_query_prompt.format(
            research_topic=research_topic,
            existing_findings=findings,
        )
        
        query_response = await search_query_model.ainvoke([
            HumanMessage(content=query_prompt)
        ])
        
        # 2. 执行搜索
        search_results = await execute_search(query_response.search_queries)
        
        # 3. 处理结果
        for result in search_results:
            processed = process_search_result(result)
            findings.append(processed)
        
        # 4. 评估是否继续
        continue_prompt = evaluate_continue_prompt.format(
            research_topic=research_topic,
            findings=findings,
        )
        
        continue_response = await continue_model.ainvoke([
            HumanMessage(content=continue_prompt)
        ])
        
        if not continue_response.should_continue:
            break
            
        search_iteration += 1
    
    return {"findings": findings}
```

### 性能分析

- **时间复杂度**：O(n × m)，n 为迭代次数，m 为每次搜索的结果数
- **空间复杂度**：O(f)，f 为研究发现数量
- **优势**：能够逐步深入，适合复杂研究任务
- **局限**：可能产生冗余信息，需要结果压缩

## 算法三：结果压缩算法

### 算法概述

当多个研究单元并发执行完成后，需要将它们的发现整合和压缩，去除重复信息，形成有序的研究发现。

**通俗解释**：这就像把多个人的笔记整理成一本书。需要去除重复的内容，按照逻辑顺序组织，形成一个连贯的叙述。

### 算法原理

```
输入：多个研究单元的输出（research_unit_outputs）
输出：压缩后的研究发现（compressed_findings）

1. 收集所有研究发现
2. 去除重复内容
3. 按主题分组
4. 按逻辑顺序排列
5. 生成摘要
6. 返回压缩后的研究发现
```

### 算法实现

关键代码位于 `compress_research` 函数中：

```python
async def compress_research(state: SupervisorState, config: RunnableConfig):
    # 1. 收集所有研究发现
    all_findings = []
    for unit_output in state["research_unit_outputs"]:
        all_findings.extend(unit_output["findings"])
    
    # 2. 使用模型压缩和整合
    compress_prompt = compress_research_system_prompt.format(
        research_brief=state["research_brief"],
        findings=all_findings,
    )
    
    compress_model = configurable_model.with_config({
        "model": configurable.compression_model,
        "max_tokens": configurable.compression_model_max_tokens,
    })
    
    compressed = await compress_model.ainvoke([
        HumanMessage(content=compress_prompt)
    ])
    
    return {
        "research_findings": compressed.compressed_findings,
    }
```

### 性能分析

- **时间复杂度**：O(n × log n)，n 为研究发现数量
- **空间复杂度**：O(n)，n 为研究发现数量
- **优势**：去除冗余，形成有序结构
- **局限**：依赖模型的压缩能力

## 算法四：结构化输出算法

### 算法概述

系统大量使用 Pydantic 模型进行结构化输出，确保模型返回的数据符合预定义的结构，便于后续处理。

**通俗解释**：这就像填写表格。表格预先定义好了字段（名字、日期、内容等），你需要按照表格的格式填写。这样无论谁来填，内容都是统一的格式，便于后续处理。

### 算法原理

```
输入：自然语言响应
输出：结构化对象（Pydantic 模型）

1. 定义 Pydantic 模型类，指定字段和类型
2. 使用 with_structured_output() 方法包装模型
3. 调用模型，传入自然语言提示
4. 模型返回结构化的对象
5. 访问对象的属性获取数据
```

### 算法实现

定义 Pydantic 模型：

```python
from pydantic import BaseModel, Field
from typing import Literal

class ClarifyWithUser(BaseModel):
    """澄清分析结果"""
    need_clarification: bool = Field(
        description="Whether the user's request needs clarification"
    )
    question: str = Field(
        description="Clarifying question if needed"
    )
    verification: str = Field(
        description="Verification message if clarification not needed"
    )

class ResearchQuestion(BaseModel):
    """研究问题结构"""
    research_brief: str = Field(
        description="The research brief/topic"
    )
    research_questions: list[str] = Field(
        description="List of research questions to investigate"
    )
    verification: str = Field(
        description="Verification message"
    )
```

使用结构化输出：

```python
# 创建结构化模型
clarification_model = (
    configurable_model
    .with_structured_output(ClarifyWithUser)
    .with_retry(stop_after_attempt=3)
)

# 调用模型
response = await clarification_model.ainvoke([
    HumanMessage(content=prompt)
])

# 访问结构化属性
if response.need_clarification:
    question = response.question
else:
    verification = response.verification
```

### 性能分析

- **时间复杂度**：O(1)，结构化不增加额外时间复杂度
- **空间复杂度**：O(1)，结构化不增加额外空间复杂度
- **优势**：确保数据一致性，便于处理
- **局限**：需要预定义模型结构

## 算法五：并发执行算法

### 算法概述

系统支持多个研究单元并发执行，通过配置 `max_concurrent_research_units` 参数控制并发数量。

**通俗解释**：这就像同时派多个调查员去不同的地方搜集信息。所有调查员同时工作，最后把大家的结果汇总。这样比一个一个调查要快得多。

### 算法原理

```
输入：研究任务列表
输出：研究结果列表

1. 将研究任务分成多个批次
2. 每批最多 max_concurrent_research_units 个任务
3. 并发执行每批任务
4. 等待所有任务完成
5. 汇总所有结果
```

### 算法实现

关键代码位于 `conduct_research` 函数中的并发部分：

```python
# 并发执行多个研究单元
research_tasks = []
for i in range(num_research_units):
    task = conduct_research_unit(
        research_unit_id=i,
        research_topic=research_topic,
        config=config,
    )
    research_tasks.append(task)

# 使用 asyncio.gather 并发执行
results = await asyncio.gather(*research_tasks)

# 汇总结果
all_findings = []
for result in results:
    all_findings.extend(result["findings"])
```

### 性能分析

- **时间复杂度**：O(n/k)，n 为任务数，k 为并发数
- **空间复杂度**：O(k)，k 为并发数
- **优势**：大幅提高执行效率
- **局限**：可能触发 API 速率限制
