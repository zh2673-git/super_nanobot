# Open Deep Research 架构分析

## 核心概念

- **[LangGraph](LangGraph.md)**：工作流编排框架，像"指挥官"一样管理整个研究流程的各个阶段
- **[StateGraph](StateGraph.md)**：状态图数据结构，像"地图"一样定义研究流程的节点和路径
- **[AgentState](AgentState.md)**：代理状态，像"记事本"一样记录研究过程中的所有信息
- **[Configuration](Configuration.md)**：配置管理，像"控制面板"一样控制代理的行为
- **[Supervisor-Agent](Supervisor-Agent.md)**：监督-执行模式，像"项目经理"分配任务给"研究员"

## 组件关系

### 1. 澄清模块 (clarify_with_user)

这个模块像"前台接待员"，负责分析用户的问题，判断是否需要进一步澄清。当用户的问题过于模糊时，它会提出澄清问题；当问题清晰时，它会生成验证消息并进入下一阶段。

### 2. 研究简报模块 (write_research_brief)

这个模块像"需求分析师"，负责将用户的原始消息转换成结构化的研究简报（ResearchQuestion），为后续研究提供明确的目标和方向。

### 3. 研究监督模块 (research_supervisor)

这是整个系统的"大脑"和"指挥官"。它使用 Supervisor-Agent 模式：
- **监督者（Supervisor）**：负责任务规划和分配，决定下一步做什么
- **研究者（Researcher）**：执行具体的研究任务，调用搜索工具获取信息

监督者会循环调用研究者，直到研究目标达成或达到最大迭代次数。

### 4. 研究执行模块 (conduct_research)

这是实际执行搜索和收集信息的"工人"。每个研究单元可以独立运行，支持并发执行以提高效率。

### 5. 结果压缩模块 (compress_research)

这个模块像"整理师"，负责将多个研究单元的结果进行压缩和整合，去除重复信息，形成有序的研究发现。

### 6. 最终报告生成模块 (generate_final_report)

这个模块像"作家"，根据压缩后的研究发现，生成结构化的最终研究报告。

## 状态管理

系统使用 Pydantic 模型定义状态结构：

- **AgentState**：整个系统的全局状态
- **ResearcherState**：研究单元的局部状态
- **SupervisorState**：监督者的状态

状态沿着图的边流动，每个节点都可以读取和修改状态。

## 配置驱动

整个系统高度可配置，通过 Configuration 类管理：

```python
# 核心配置项
max_concurrent_research_units = 5   # 最大并发研究单元数
max_researcher_iterations = 6      # 最大研究迭代次数
search_api = SearchAPI.TAVILY       # 搜索 API 选择
```

## 扩展性

系统支持多种扩展方式：

1. **MCP 服务器**：通过 Model Context Protocol 添加外部工具
2. **自定义搜索 API**：实现搜索工具接口
3. **自定义模型**：支持任何 LangChain 支持的聊天模型
4. **自定义提示词**：修改 prompts.py 中的模板
