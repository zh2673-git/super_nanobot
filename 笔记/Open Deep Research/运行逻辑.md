# Open Deep Research 运行逻辑

## 核心概念

- **[入口点](入口点.md)**：程序的启动位置，像"大门"一样进入整个研究流程
- **[状态流转](状态流转.md)**：数据在各个模块之间传递的过程，像"流水线"一样一步步处理
- **[节点函数](节点函数.md)**：每个处理单元，像"工人"一样完成特定任务
- **[条件跳转](条件跳转.md)**：根据状态决定下一步走向，像"岔路口"的选择

## 程序执行流程

### 第一阶段：用户输入处理

```
[用户发送消息]
    │
    ▼
[clarify_with_user 节点]
    │
    ├─需要澄清？──是──→ [结束：返回澄清问题]
    │
    └─否──→ [write_research_brief 节点]
              │
              ▼
        [生成研究简报]
              │
              ▼
```

**入口函数 clarify_with_user**：这个函数像"前台接待员"，它的职责是：
1. 接收用户的原始消息
2. 分析消息是否足够清晰
3. 如果不清晰，生成澄清问题询问用户
4. 如果清晰，生成验证消息并进入下一阶段

**入口函数 write_research_brief**：这个函数像"需求分析师"，它的职责是：
1. 将用户的模糊问题转换成结构化的研究目标
2. 生成清晰的研究简报（ResearchQuestion）
3. 准备传递给监督者的上下文

### 第二阶段：研究监督循环

```
[research_supervisor 节点]
    │
    ▼
[Supervisor 决策]
    │
    ├─需要更多研究？──是──→ [分配研究任务]
    │                            │
    │                            ▼
    │                     [conduct_research 节点]
    │                            │
    │                            ▼
    │                     [返回研究发现]
    │                            │
    └────────────────────────────┘
              │
              ▼
        [compress_research 节点]
              │
              ▼
        [检查是否继续]
              │
              ├─继续迭代──→ [返回 Supervisor 决策]
              │
              └─结束──→ [generate_final_report 节点]
```

**核心循环说明**：
1. **Supervisor 决策**：监督者分析当前研究进度，决定是否需要继续研究
2. **分配研究任务**：将研究目标分解为多个子任务，分配给不同的研究单元
3. **执行研究**：每个研究单元独立执行搜索和信息收集
4. **压缩结果**：将多个研究单元的结果整合压缩
5. **检查迭代**：判断是否达到研究目标或最大迭代次数
6. **生成报告**：研究完成后，生成最终报告

### 第三阶段：结果输出

```
[generate_final_report 节点]
    │
    ▼
[整合研究发现]
    │
    ▼
[撰写最终报告]
    │
    ▼
[返回给用户]
```

## 关键状态字段

### AgentState（全局状态）

```python
class AgentState(TypedDict):
    messages: list[BaseMessage]           # 对话历史
    research_brief: str                   # 研究简报
    research_plan: str                    # 研究计划
    research_findings: ResearchFindings   # 研究发现
    research_iterations: int              # 研究迭代次数
    final_report: str                     # 最终报告
```

### ResearcherState（研究单元状态）

```python
class ResearcherState(TypedDict):
    research_iteration: int               # 当前迭代次数
    search_results: list[SearchResult]    # 搜索结果
    findings: list[str]                   # 发现列表
    completed_research_units: list[str]   # 已完成的研究单元
    research_unit_outputs: dict           # 研究单元输出
```

## 搜索执行流程

每个研究单元内部也有一个执行流程：

```
[接收研究任务]
    │
    ▼
[生成搜索查询]
    │
    ▼
[调用搜索 API]
    │
    ▼
[处理搜索结果]
    │
    ▼
[评估是否需要继续搜索]
    │
    ├─是──→ [生成新查询，返回"搜索"状态]
    │
    └─否──→ [返回研究发现]
```

## 并发执行

系统支持多个研究单元并发执行：

```python
# 配置最大并发数
max_concurrent_research_units = 5
```

这意味着最多可以同时有 5 个研究单元在执行搜索任务，大大提高了研究效率。

## 错误处理

系统有完善的错误处理机制：

1. **模型重试**：如果结构化输出失败，自动重试最多 3 次
2. **Token 限制**：检测是否超过模型的 token 限制，及时压缩上下文
3. **搜索失败**：搜索 API 调用失败时，记录错误并继续执行
4. **状态恢复**：使用 LangGraph 的持久化机制，可以在失败后恢复状态

## 配置参数

关键的配置参数控制着执行流程：

| 参数 | 默认值 | 说明 |
|------|-------|------|
| max_researcher_iterations | 6 | 最大研究迭代次数 |
| max_concurrent_research_units | 5 | 最大并发研究单元数 |
| allow_clarification | true | 是否允许澄清问题 |
| search_api | TAVILY | 搜索 API 选择 |
