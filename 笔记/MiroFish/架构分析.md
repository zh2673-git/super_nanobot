# MiroFish 架构分析

## 🎯 第一步：点出核心架构组件

在深入理解架构之前，让我们先认识构成这个系统的核心"积木块"：

- **[Flask后端]：Web服务框架（就像"餐厅服务员"，接收用户订单，把需求传递给厨房）**
- **[OASIS模拟引擎]：社交媒体模拟核心（像"游戏引擎"，驱动虚拟角色的行为）**
- **[Zep图谱服务]：知识图谱存储与检索（像"超级大脑"，存储实体之间的关系）**
- **[LLM客户端]：AI大模型调用封装（像"翻译官"，把用户需求翻译给AI理解）**
- **[项目管理系统]：状态持久化（像"账本"，记录每个项目的进度和状态）**
- **[Vue.js前端]：用户交互界面（像"收银台"，展示菜品、接收点餐）**

**架构分层说明**：

MiroFish的架构就像一个现代化的"AI工厂"：

1. **前端展示层（收银台）**：Vue.js构建的用户界面，负责展示模拟世界、接收用户输入
2. **API路由层（服务员）**：Flask路由负责接收HTTP请求，把"订单"分发给不同的处理函数
3. **服务层（厨师团队）**：每个Service是一个专业厨师：
   - OntologyGenerator：负责"消化"原材料（生成知识本体）
   - GraphBuilderService：负责"搭建骨架"（构建知识图谱）
   - SimulationManager：负责"导演"角色（管理模拟生命周期）
   - OasisProfileGenerator：负责"创造角色"（生成Agent人设）
   - ReportAgent：负责"写报告"（分析模拟结果）

4. **模型层（账本）**：记录每个项目、任务、模拟的状态，确保断点可续
5. **外部服务（原材料供应商）**：
   - LLM：提供AI思考能力
   - Zep Cloud：提供知识图谱存储和检索
   - OASIS：提供社交媒体模拟引擎

## 📖 第三步：详细架构组件解析

### 1. Flask 后端架构

后端采用标准的**分层架构**，代码组织在 `backend/app/` 目录下：

```
backend/app/
├── api/                    # API 路由层（接收请求）
│   ├── graph.py           # 图谱构建相关接口
│   ├── simulation.py       # 模拟运行相关接口
│   ├── report.py           # 报告生成相关接口
│   └── __init__.py
│
├── services/               # 服务层（业务逻辑）
│   ├── ontology_generator.py      # 本体生成服务
│   ├── graph_builder.py           # 图谱构建服务
│   ├── zep_entity_reader.py       # 实体读取服务
│   ├── oasis_profile_generator.py # Agent人设生成
│   ├── simulation_manager.py      # 模拟生命周期管理
│   ├── simulation_runner.py       # 模拟运行器
│   └── report_agent.py            # 报告生成代理
│
├── models/                 # 模型层（数据结构）
│   ├── project.py          # 项目数据模型
│   ├── task.py            # 任务数据模型
│   └── __init__.py
│
├── utils/                  # 工具函数
│   ├── llm_client.py      # LLM调用封装
│   ├── file_parser.py     # 文件解析（PDF/TXT）
│   ├── logger.py          # 日志工具
│   ├── retry.py           # 重试机制
│   └── zep_paging.py      # Zep分页工具
│
├── config.py               # 配置管理
└── __init__.py            # Flask应用工厂
```

**通俗理解**：这种分层就像一家餐厅的运营：
- **API层** = 服务员，负责接待客人（HTTP请求）
- **Services层** = 厨房里的各个厨师，每个厨师有专长
- **Models层** = 账本，记录各种状态
- **Utils层** = 厨房里的锅碗瓢盆、刀具等工具

### 2. API 路由设计

MiroFish的API设计遵循**RESTful风格**，主要分为三大模块：

#### 2.1 图谱构建API (`/api/graph`)

| 接口 | 方法 | 功能 |
|-----|------|-----|
| `/ontology/generate` | POST | 上传文档，生成本体定义 |
| `/build` | POST | 构建知识图谱 |
| `/task/<task_id>` | GET | 查询构建进度 |
| `/data/<graph_id>` | GET | 获取图谱数据 |
| `/project/<project_id>` | GET/DELETE | 项目管理 |

**工作流程**：
```
用户上传文档 
    → /ontology/generate (调用LLM生成实体类型、关系类型)
    → 返回 project_id
    → /build (异步构建图谱，存储到Zep)
    → 返回 task_id
    → 查询 task_id 获取进度
    → 完成
```

#### 2.2 模拟运行API (`/api/simulation`)

| 接口 | 方法 | 功能 |
|-----|------|-----|
| `/create` | POST | 创建新模拟 |
| `/prepare` | POST | 准备模拟环境（生成人设、配置） |
| `/entities/<graph_id>` | GET | 读取图谱实体 |
| `/run` | POST | 启动模拟运行 |
| `/list` | GET | 列出所有模拟 |

**工作流程**：
```
项目已有图谱 
    → /create (创建模拟，返回 simulation_id)
    → /prepare (异步生成Agent人设和配置)
    → /run (运行OASIS模拟)
    → 获取模拟结果
```

#### 2.3 报告生成API (`/api/report`)

| 接口 | 方法 | 功能 |
|-----|------|-----|
| `/generate` | POST | 生成预测报告 |
| `/interview` | POST | 与模拟世界交互 |

### 3. 服务层详解

#### 3.1 OntologyGenerator（本体生成服务）

**职责**：理解用户上传的文档，提取核心概念

**核心方法**：
```python
def generate(document_texts, simulation_requirement, additional_context):
    """
    输入：文档文本列表 + 模拟需求
    输出：本体定义（实体类型列表 + 关系类型列表 + 分析摘要）
    """
```

**通俗理解**：就像阅读理解题目中的"总结中心思想"，LLM会分析文档，告诉我们：
- 文档涉及哪些"人物"（实体类型）：如学生、老师、网民
- 这些人物之间是什么"关系"（关系类型）：如师生关系、关注关系
- 文档大概在讲什么（分析摘要）

#### 3.2 GraphBuilderService（图谱构建服务）

**职责**：将文本切块，存储到Zep图谱服务

**核心方法**：
```python
def create_graph(name):              # 创建图谱
def set_ontology(graph_id, ontology) # 设置本体定义
def add_text_batches(...)            # 批量添加文本
def get_graph_data(graph_id)         # 获取图谱数据
```

**通俗理解**：就像把一本书"拆解"并"索引"：
- **文本切块**：把长文档切成500字符的小段落（方便检索）
- **添加本体**：告诉Zep"这本书的人物关系是这样的"
- **存储索引**：把切好的段落存到Zep，建立关键词索引

#### 3.3 SimulationManager（模拟管理器）

**职责**：管理模拟的整个生命周期

**核心状态**：
- CREATED（创建）
- PREPARING（准备中）
- READY（就绪）
- RUNNING（运行中）
- COMPLETED（完成）
- FAILED（失败）

**通俗理解**：就像一个"项目经理"，负责协调模拟的各个环节：
1. 协调各个环节的进度
2. 存储中间产物（人设、配置）
3. 提供运行脚本

#### 3.4 OasisProfileGenerator（Agent人设生成）

**职责**：为每个实体生成"人设卡片"

**核心逻辑**：
```
实体信息 → LLM生成 → OASIS Profile格式
   ↓
姓名、头像、性格、立场、说话风格...
```

**通俗理解**：就像《演员请就位》中，导演根据剧本角色生成详细的人物小传：
- 这个角色叫什么名字？
- 他的性格是内向还是外向？
- 他会支持还是反对？
- 他平时会发什么样的帖子？

#### 3.5 ReportAgent（报告代理）

**职责**：分析模拟结果，生成预测报告

**核心能力**：
- 拥有多种工具（搜索帖子、查询用户、分析趋势）
- 可以深度"采访"模拟世界
- 生成结构化的预测报告

**通俗理解**：就像一个"资深记者"，在模拟世界里"采访"了一圈后，写出一篇专业的分析报告。

### 4. 数据模型设计

#### 4.1 Project（项目模型）

记录每个用户项目的基本信息：
- project_id：项目唯一标识
- name：项目名称
- simulation_requirement：模拟需求
- ontology：本体定义
- graph_id：关联的图谱ID
- status：项目状态
- files：上传的文件列表

#### 4.2 Task（任务模型）

记录异步任务的执行状态：
- task_id：任务ID
- status：pending/processing/completed/failed
- progress：0-100
- message：进度描述
- result：任务结果

#### 4.3 Simulation（模拟状态）

记录每次模拟的详细状态：
- simulation_id：模拟ID
- project_id：关联项目
- graph_id：图谱ID
- status：模拟状态
- entity_types：涉及的实体类型
- profiles_count：生成的Agent数量

前端采用 **Vue 3 + Vue Router + Pinia** 构建：

```

│   ├── Home.vue          # 首页（历史项目）
│   ├── MainView.vue      # 主流程页面

│   ├── SimulationRunView.vue # 模拟运行
│   └── ReportView.vue    # 报告查看
│
├── components/           # 可复用组件
│   ├── Step1GraphBuild.vue   # 步骤1：图谱构建
│   ├── Step2EnvSetup.vue     # 步骤2：环境搭建
│   ├── Step3Simulation.vue   # 步骤3：模拟运行
│   ├── Step4Report.vue       # 步骤4：报告生成
│   ├── Step5Interaction.vue # 步骤5：深度互动
│   ├── GraphPanel.vue        # 图谱可视化

│
├── api/                  # API调用封装

│   ├── simulation.js     # 模拟API
│   └── report.js         # 报告API
│
├── store/                # Pinia状态管理
```

**五步工作流**：
1. Step1：上传文档 → 生成图谱
2. Step2：读取实体 → 生成Agent人设

4. Step4：ReportAgent分析生成报告
5. Step5：与模拟世界深度互动

### 7. 异步任务处理

MiroFish使用 **Python线程** 处理异步任务：

```python
# 创建后台线程
thread = threading.Thread(target=build_task, daemon=True)
thread.start()

# 返回task_id供前端轮询
return jsonify({
    "task_id": task_id,
    "message": "任务已启动，请查询进度"
})
```

**通俗理解**：就像餐厅接单后，服务员把订单交给厨房，然后给顾客一个"取餐号"，顾客可以随时查询菜做好了没有。
