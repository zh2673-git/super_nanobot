# MiroFish 运行逻辑

## 🎯 第一步：点出核心运行流程

在深入理解程序如何运行之前，让我们先认识它的核心"流水线"：

- **[图谱构建流程]：把原始文档变成可查询的知识网络（像把原材料加工成产品）**
- **[环境准备流程]：读取知识图谱，生成虚拟角色（像根据剧本选演员、定人设）**
- **[模拟运行流程]：驱动Agent在虚拟社交媒体上互动（像开机拍摄电影）**
- **[报告生成流程]：分析模拟结果，输出预测报告（像电影杀青后写影评）**
- **[互动交流流程]：与模拟世界中的任意角色对话（像与电影角色实时对话）**

## 📖 第三步：详细运行逻辑解析

### 1. 本体生成流程（OntologyGenerator）

**核心目标**：理解用户上传的文档，提取"谁在文中出现，他们是什么关系"

**详细步骤**：

```
输入：
- document_texts: 文档文本列表
- simulation_requirement: "预测这个事件会如何发酵"
- additional_context: 额外说明

处理过程：
1. 合并文本（最多5万字）
2. 构建Prompt（系统提示+用户消息）
3. 调用LLM（GPT-4等）
4. 解析JSON响应
5. 后处理验证

输出（本体定义）：
{
    "entity_types": [    // 正好10个
        {name: "Student", description: "...", attributes: [...]},
        {name: "Professor", ...},
        ...
        {name: "Person", ...},     // 兜底
        {name: "Organization", ...} // 兜底
    ],
    "edge_types": [      // 6-10个
        {name: "STUDIES_AT", source_targets: [...]},
        {name: "WORKS_FOR", ...},
        ...
    ],
    "analysis_summary": "..."
}
```

**通俗理解**：就像读一本小说后，总结出：
- 书中有哪些角色（实体类型）：主角、配角、老师、学生、公司
- 角色之间什么关系（关系类型）：师生关系、雇佣关系
- 故事大概讲什么（分析摘要）

### 2. 图谱构建流程（GraphBuilderService）

**核心目标**：把文本存到Zep图谱服务，建立可检索的知识网络

**详细步骤**：

```
输入：
- project_id: 项目ID
- ontology: 本体定义
- text: 提取的文本

处理过程：
1. 文本分块
   - chunk_size = 500字符
   - overlap = 50字符
   - 例如："今天天气很好..." → ["今天天气很好...", "气很好...小明"]

2. 创建图谱
   - 调用Zep API: create_graph(name)
   - 返回graph_id

3. 设置本体
   - 调用Zep API: set_ontology(graph_id, entity_types, edge_types)

4. 批量添加文本
   - 分批添加（每批3个）
   - 每批调用Zep API: add_documents()
   - 记录episode_uuid

5. 等待处理
   - 轮询episode状态
   - 等待Zep完成索引

6. 获取图谱数据
   - 调用Zep API: get_graph()
   - 返回节点和边的数量
```

**通俗理解**：就像把一本书变成"可搜索的数据库"：
- 把书切成小段落（分块）
- 告诉数据库"这本书的人物关系"（设置本体）
- 把段落存进去，建立索引（添加文本）
- 可以按关键词快速找到相关内容

### 3. 实体读取与过滤（ZepEntityReader）

**核心目标**：从图谱中提取实体，为生成Agent做准备

**详细步骤**：

```
输入：
- graph_id: 图谱ID
- entity_types: 要过滤的实体类型列表

处理过程：
1. 获取所有节点
   - 调用Zep API: graph.get_nodes()

2. 过滤实体
   - 筛选出Labels包含预定义实体类型的节点
   - 例如：只保留类型为"Student"的节点

3. 获取上下文
   - 对每个实体，获取相关的边信息
   - 获取实体描述
   - 获取记忆内容

输出：
{
    "entity_types": ["Student", "Professor", ...],
    "entities": [
        {
            "uuid": "...",
            "name": "张三",
            "type": "Student",
            "attributes": {...},
            "edges": [...],
            "memory": "..."
        },
        ...
    ]
}
```

**通俗理解**：就像从"人才数据库"中筛选出符合条件的人：
- 找出所有学生
- 找出所有教授
- 记录每个人的背景、经历、社会关系

### 4. Agent人设生成（OasisProfileGenerator）

**核心目标**：为每个实体生成"人设卡片"，让AI可以扮演这个角色

**详细步骤**：

```
输入：
- entities: 实体列表
- simulation_requirement: 模拟需求
- document_text: 原始文档
- platform: reddit/twitter

处理过程（每个实体）：
1. 构建Prompt
   - 实体信息（姓名、类型、属性）
   - 实体在图谱中的上下文
   - 模拟需求
   - 平台特点

2. 调用LLM生成人设
   - 姓名（保持或稍作修改）
   - 用户名（符合平台风格）
   - 头像描述
   - 性格特点
   - 立场（支持/反对/中立）
   - 发言风格
   - 背景故事

3. 转换为OASIS格式
   - Reddit: JSON格式
   - Twitter: CSV格式

输出：
- reddit_profiles.json: [
    {
        "agent_name": "张三",
        "username": "student_zhang",
        "platform": "reddit",
        "persona": "...",
        ...
    },
    ...
  ]
```

**通俗理解**：就像为一个电影角色写"人物小传"：
- 这个角色叫什么、什么性格
- 他支持还是反对这件事
- 他平时会说什么样的话
- 他有什么背景故事

### 5. 模拟配置生成（SimulationConfigGenerator）

**核心目标**：LLM智能生成模拟参数

**详细步骤**：

```
输入：
- simulation_requirement: 模拟需求
- entities_count: 实体数量
- entity_types: 实体类型
- document_text: 原始文档
- platform: twitter/reddit

处理过程：
1. 分析需求和实体
2. 调用LLM生成配置

输出（simulation_config.json）：
{
    "time_config": {
        "total_simulation_hours": 24,   // 模拟24小时
        "minutes_per_round": 30,       // 每轮30分钟
        "peak_hours": [9, 12, 18, 21], // 高峰时段
        "trough_hours": [2, 3, 4, 5]   // 低谷时段
    },
    "agent_configs": [
        {
            "agent_name": "张三",
            "activity_level": "high",
            "post_frequency": "frequent",
            "stance": "support",
            ...
        },
        ...
    ],
    "event_config": {
        "initial_posts": [...],    // 初始帖子
        "hot_topics": [...]        // 热点话题
    },
    "platform_config": {...},
    "generation_reasoning": "..."  // LLM的配置理由
}
```

**通俗理解**：就像制定一个"拍摄计划"：
- 模拟多长时间（时间配置）
- 谁在什么时候活跃（Agent配置）
- 有什么事件发生（事件配置）
- 每个平台有什么规则（平台配置）

### 6. 模拟运行（SimulationRunner）

**核心目标**：实际运行OASIS模拟引擎

**详细步骤**：

```
输入：
- simulation_id
- max_rounds: 最大轮数
- platforms: [twitter, reddit]

处理过程：
1. 加载配置
   - 读取simulation_config.json
   - 读取reddit_profiles.json
   - 读取twitter_profiles.csv

2. 初始化OASIS
   - 加载Agent
   - 设置环境

3. 主循环（每轮）：
   For round in range(max_rounds):
       - Twitter平台执行
         * 获取当前状态
         * 每个Agent决定动作
         * 执行动作（发推、点赞、转发）
         * 更新状态
         
       - Reddit平台执行
         * 类似Twitter
         
       - 记录日志
       - 更新进度

4. 保存结果
   - 帖子记录
   - 互动记录
   - 最终状态
```

**通俗理解**：就像"拍摄电影"：
- 导演（SimulationRunner）指挥
- 演员（Agent）按照人设表演
- 每轮相当于"一集"
- 最后剪辑成完整的"电影"（模拟数据）

### 7. 报告生成（ReportAgent）

**核心目标**：分析模拟数据，生成预测报告

**详细步骤**：

```
输入：
- simulation_id
- requirement: 用户的问题

处理过程：
1. 初始化ReportAgent
   - 加载工具集（搜索、查询、统计）
   - 设置系统提示

2. 分析模拟结果
   - 统计各方的发言数量
   - 分析观点分布
   - 识别关键事件
   - 追踪话题演变

3. 回答用户问题
   - 结合模拟结果和LLM推理
   - 生成结构化报告

输出：
- Markdown格式的预测报告
- 包含：事件概述、各方反应、趋势预测、关键节点
```

**通俗理解**：就像"影评人写影评"：
- 观看完整个电影（分析模拟数据）
- 回答观众的问题（回答用户询问）
- 写出专业的分析报告（预测报告）

## 🔄 状态机转换

### Project状态

```
CREATED ──(ontology生成)──▶ ONTOLOGY_GENERATED ──(图谱构建)──▶ GRAPH_BUILDING
                                                                            │
                                                                            ▼
                                                                    GRAPH_COMPLETED
                                                                            │
                                                                            ▼
                                                                       (可创建模拟)
```

### Simulation状态

```
CREATED ──(prepare)──▶ PREPARING ──(完成)──▶ READY ──(run)──▶ RUNNING
                                                                       │
                                          ┌────────────────────────────┘
                                          │
                                          ▼
                              COMPLETED / STOPPED / FAILED
```

### Task状态

```
PENDING ──(开始执行)──▶ PROCESSING ──(完成)──▶ COMPLETED
                          │
                          └─(失败)──▶ FAILED
```
