# Local Deep Research 架构分析

> 本笔记深入分析项目的系统架构，帮助理解各个组件如何协作。

---

## 第一步：点出架构的核心组件概念

- **[Web服务层（Flask Web Server）]：面向用户的"门面"，处理网页请求、用户登录、API调用，就像餐厅的"服务员"**
- **[研究引擎（Research Engine）]：项目的"大脑"，负责协调整个研究流程，就像餐厅的"主厨"**
- **[搜索引擎抽象层（Search Abstraction Layer）]：统一的"搜索接口"，让项目可以灵活使用20+种搜索引擎，就像"万能转接头"**
- **[LLM编排层（LLM Orchestration）]：AI模型的"调度中心"，管理不同AI模型的调用和切换，就像"采购员"负责挑选合适的"食材"**
- **[数据持久化层（Data Persistence）]：数据的"仓库管理员"，负责存储和检索所有数据，包括加密的研究历史和知识库**

---

## 第二步：用图构建架构组件之间的联系

```
                                    [用户浏览器/客户端]
                                            |
                                            v
    +------------------------------------Web服务层（Flask）------------------------------------+
    |  [认证模块] <-> [路由处理器] <-> [WebSocket服务] <-> [静态资源]                         |
    +---------------------------------------------------------------------------------------+
                                            |
                    +-----------------------+-----------------------+
                    |                       |                       |
                    v                       v                       v
    [研究引擎]            [知识库系统]              [基准测试模块]
        |                       |                       |
        +---------------+--------+-----------+-----------+
                        |                        |
                        v                        v
    +--------搜索引擎抽象层--------+    +-----LLM编排层-----+
    | [搜索引擎工厂] [搜索引擎基类] |    | [Provider接口]     |
    | [SearXNG] [arXiv] [PubMed]   |    | [OpenAI] [Claude] |
    +------------------------------+    | [Ollama] [Google]  |
                                        +-------------------+
                                                |
                                                v
                                +------------数据持久化层------------+
                                | [SQLCipher加密数据库] [FAISS索引]   |
                                +------------------------------------+
```

**联系说明**：整个系统像一个现代化餐厅的运作：
1. **服务员（Web服务层）** 接收顾客（用户）的点餐（研究请求）
2. **主厨（研究引擎）** 协调整个烹饪过程（研究过程）
3. **采购员（LLM编排层）** 选择合适的食材（AI模型）
4. **万能转接头（搜索引擎抽象层）** 从不同供应商获取原料（搜索结果）
5. **仓库管理员（数据持久化层）** 管理食材库存（存储数据）

## 第三步：详细解释架构组件

### 1. Web服务层（Flask Web Server）

**专业定义**：基于Flask框架构建的Web服务，提供用户界面、REST API、WebSocket实时通信等功能。

**通俗解释与类比**：就像餐厅的"服务员"团队：
- 负责接收顾客的点餐（用户请求）
- 把订单传给厨房（后端处理）
- 把做好的菜端给顾客（返回结果）
- 还负责处理顾客的身份验证（登录/注册）

**在此项目中的具体体现**：
```python
# 核心入口
src/local_deep_research/web/app.py

# 主要功能模块
- 用户认证 (Flask-Login)
- REST API (Flask-RESTful)
- WebSocket实时更新 (Flask-SocketIO)
- 速率限制 (Flask-Limiter)
```

---

### 2. 研究引擎（Research Engine）

**专业定义**：核心研究逻辑所在，负责接收研究请求、制定研究计划、执行搜索迭代、分析结果并生成报告。

**通俗解释与类比**：就像餐厅的"主厨"：
- 接收"点餐"（研究请求）
- 制定"烹饪计划"（研究策略）
- 协调"备菜"（搜索和获取资料）
- 最后的"烹饪和摆盘"（生成报告）

**在此项目中的具体体现**：
```python
# 研究服务
src/local_deep_research/web/services/research_service.py

# 研究模式
- quick_summary: 快速摘要（30秒-3分钟）
- detailed_research: 详细研究（5-15分钟）
- report_generation: 报告生成（10-30分钟）
- document_analysis: 文档分析

# 研究策略（20+种）
- focused_iteration: 聚焦迭代
- broad_coverage: 广泛覆盖
- academic_search: 学术搜索
- ...
```

---

### 3. 搜索引擎抽象层（Search Abstraction Layer）

**专业定义**：模块化的搜索引擎框架，通过抽象基类（SearchEngineBase）定义统一接口，让每种搜索引擎实现自己的搜索逻辑。

**通俗解释与类比**：就像一个"万能转接头"：
- 不管是"英标插头"（Google）还是"美标插头"（arXiv）
- 都能转换成"统一接口"（项目内部的SearchResult格式）
- 用户和上层代码不需要关心底层具体是哪个搜索引擎

**核心抽象**：
```python
# 搜索引擎基类
src/local_deep_research/web_search_engines/search_engine_base.py

class SearchEngineBase:
    def run(self, query: str) -> List[SearchResult]:
        """执行搜索，返回结果列表"""
        pass
```

**支持的搜索引擎（部分）**：
| 文件 | 搜索引擎 | 用途 |
|-----|---------|------|
| search_engine_searxng.py | SearXNG | 元搜索，综合结果 |
| search_engine_arxiv.py | arXiv | 学术论文 |
| search_engine_pubmed.py | PubMed | 医学文献 |
| search_engine_google_pse.py | Google | 综合搜索 |
| search_engine_tavily.py | Tavily | AI优化搜索 |
| search_engine_wikipedia.py | Wikipedia | 百科知识 |
| search_engine_github.py | GitHub | 代码搜索 |

---

### 4. LLM编排层（LLM Orchestration）

**专业定义**：基于LangChain的LLM集成框架，支持多种AI模型provider的统一接口。

**通俗解释与类比**：就像餐厅的"采购员"：
- 根据"菜谱"（研究需求）选择合适的"食材"（AI模型）
- 有的菜需要"进口牛肉"（GPT-4，能力强但贵）
- 有的菜用"本地猪肉"（Ollama，便宜但能力有限）
- 采购员负责挑选最合适的

**支持的LLM Provider**：
```python
# 目录结构
src/local_deep_research/llm/providers/
├── implementations/
│   ├── openai.py        # OpenAI GPT系列
│   ├── anthropic.py     # Anthropic Claude系列
│   ├── google.py        # Google Gemini系列
│   ├── ollama.py        # 本地Ollama模型
│   ├── openrouter.py    # OpenRouter（100+模型）
│   └── ...
├── base.py              # Provider基类
└── llm_registry.py      # Provider注册表
```

---

### 5. 数据持久化层（Data Persistence）

**专业定义**：使用SQLCipher实现加密存储，结合FAISS实现向量相似度搜索。

**通俗解释与类比**：就像餐厅的"仓库管理系统"：
- **普通数据（研究历史、用户信息）** → 存放在"保险柜"（SQLCipher加密数据库）
- **向量数据（知识库索引）** → 存放在"智能仓库"（FAISS，可以快速找到相似内容）

**数据存储架构**：
```
用户数据
    |
    +-- [SQLCipher加密数据库]
    |   - 用户账号信息
    |   - 研究历史记录
    |   - 系统配置
    |
    +-- [FAISS向量索引]
        - 文档向量
        - 知识库内容
        - 搜索索引
```

## 扩展性设计

### 添加新的搜索引擎
```python
# 1. 继承基类
class MySearchEngine(SearchEngineBase):
    def run(self, query: str) -> List[SearchResult]:
        # 实现搜索逻辑
        pass

# 2. 注册到工厂
SearchEngineFactory.register("my_engine", MySearchEngine)
```

### 添加新的LLM Provider
```python
# 1. 继承基类
class MyLLMProvider(BaseLLMProvider):
    def generate(self, prompt: str) -> str:
        # 实现生成逻辑
        pass

# 2. 注册到注册表
LLMRegistry.register("my_provider", MyLLMProvider)
```
