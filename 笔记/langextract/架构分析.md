# LangExtract 架构分析

## 概述

LangExtract 采用分层模块化架构，将复杂的信息提取任务分解为多个独立的模块，每个模块负责特定的功能。这种设计使得系统易于扩展、维护和测试。

---

## 第一步：点出架构组件概念

LangExtract 架构由以下核心组件构成：

- **[主入口层（extraction.py）](#主入口层extractionpy)**：整个系统的"大脑"和"总调度中心"，负责协调各模块完成提取任务
- **[提示词层（prompting.py）](#提示词层promptingpy)**：构建 LLM 指令的"编剧"，设计提取任务的提示词结构
- **[推理层（inference.py）](#推理层inferencepy)**：与外部 LLM 服务通信的"外交官"，处理 API 调用和响应解析
- **[数据层（data.py & core/data.py）](#数据层datapy--coredatapy)**：定义数据结构的"设计师"，规定提取结果的格式
- **[分词层（tokenizer.py & chunking.py）](#分词层tokenizerpy--chunkingpy)**：处理长文本的"裁缝"，将大文档切分为可处理的小块
- **[验证层（schema.py & prompt_validation.py）](#验证层schemapy--prompt_validationpy)**：质量控制的"质检员"，验证提示词和提取结果
- **[提供者层（providers/）](#提供者层providers)**：支持多种 LLM 的"万能适配器"，统一不同 API 的接口
- **[可视化层（visualization.py）](#可视化层visualizationpy)**：生成报告的"画师"，创建交互式 HTML 可视化

## 第三步：架构组件详细解析

### 主入口层 extraction.py

**专业定义**：extraction.py 是 LangExtract 的核心模块，包含 `extract()` 函数，负责整个提取流程的编排和协调。

**通俗解释与类比**：主入口层就像一家餐厅的"总经理"。顾客点菜后，总经理不会自己去切菜或炒菜，而是协调采购员（tokenizer 采购原材料）、厨师（inference 烹饪）、质检员（schema 检查）等各个环节，最后把做好的菜（提取结果）端给顾客。

**在此项目中的具体体现**：

`extract()` 函数接收用户输入的参数，包括：
- `text_or_documents`：待处理的文本
- `prompt_description`：提取规则
- `examples`：few-shot 示例
- `model_id`：使用的模型
- `max_char_buffer`：每次处理的字符数
- 其他高级配置参数

函数内部流程：
1. **预处理**：验证参数，初始化各模块
2. **分块**：调用 tokenizer 将长文本切分成小块
3. **构建提示词**：调用 prompting 构建每个块的提示词
4. **并行推理**：调用 inference 并行处理多个块
5. **验证结果**：调用 schema 验证提取结果
6. **后处理**：合并结果，处理重叠，去重
7. **返回结果**：返回结构化的提取结果

---

### 提示词层 prompting.py

**专业定义**：prompting 模块负责构建发送给 LLM 的提示词，包括任务描述、few-shot 示例、输出格式要求等。

**通俗解释与类比**：prompting 就像餐厅的"菜单设计师"或"配方师"。一道菜需要什么材料、放多少盐、放多少糖，都需要精心设计。同样，提取任务需要什么样的指令、什么样的示例、什么样的输出格式，都由 prompting 模块来设计。

**在此项目中的具体体现**：

prompting 模块的核心功能：
- **任务描述构建**：将用户的 `prompt_description` 转换为结构化的任务指令
- **示例格式化**：将 `ExampleData` 转换为 few-shot 示例
- **输出格式指定**：根据 `format_type` 指定 JSON 或 YAML 输出格式
- **约束条件注入**：添加模式约束到提示词中

---

### 推理层 inference.py

**专业定义**：inference 模块负责与各种 LLM 服务进行交互，包括请求构建、API 调用、响应解析等。

**通俗解释与类比**：inference 就像餐厅的"采购员"或"外卖员"。厨房（LangExtract）需要知道某种食材的味道，但自己不会变出答案，需要派采购员去市场（LLM API）询问，然后带回答案。

**在此项目中的具体体现**：

inference 模块的核心功能：
- **请求构建**：将提示词和参数封装成 API 请求
- **API 调用**：通过 aiohttp 发起异步 HTTP 请求
- **响应解析**：解析 LLM 返回的 JSON/JSONL 响应
- **错误处理**：处理 API 错误、超时等异常情况
- **重试机制**：在失败时自动重试

---

### 分词层 tokenizer.py & chunking.py

**专业定义**：tokenizer 和 chunking 模块负责将长文档切分成适合 LLM 处理的小块，同时保持语义连贯性。

**通俗解释与类比**：tokenizer 就像"裁缝"或"分解师"。一块大布料（长文本）没法一次性塞进缝纫机，需要先剪成小块（chunks），分批处理，最后再拼接成完整的成品。

**在此项目中的具体体现**：

- **RegexTokenizer**：基于正则表达式的分词器，按句子或段落切分
- **对齐计算**：确保每个提取结果能映射回原文的精确位置
- **重叠处理**：相邻块之间有重叠，避免丢失边界信息
- **上下文保持**：保留块之间的上下文连贯性

---

### 验证层 schema.py & prompt_validation.py

**专业定义**：schema 模块负责定义和验证提取结果的模式约束；prompt_validation 模块负责验证提示词的质量。

**通俗解释与类比**：验证层就像"质检员"或"门卫"。工厂生产的产品需要符合规格，质检员会检查产品是否合格。同样，提取结果需要符合预定义的模式，验证层负责检查这一点。

**在此项目中的具体体现**：

- **模式定义**：定义 Extraction、Attribute 等数据结构的模式
- **约束类型**：支持字符串、枚举、列表、嵌套等多种约束
- **验证逻辑**：检查提取结果是否满足所有约束
- **提示词验证**：检查提示词是否符合最佳实践

---

### 数据层 data.py & core/data.py

**专业定义**：data 模块定义了 LangExtract 使用的所有核心数据结构，包括 Extraction、ExampleData、AnnotatedDocument 等。

**通俗解释与类比**：数据层就像"表格模板"或"填写说明"。就像报销需要填特定的表格，提取结果也需要按照预定义的格式来组织，这样后续处理才能统一处理这些数据。

**在此项目中的具体体现**：

核心数据结构：
- **Extraction**：单个提取结果，包含：
  - `extraction_class`：提取类别（如 "character"、"emotion"）
  - `extraction_text`：提取的文本
  - `attributes`：属性字典
  - `source_start`、`source_end`：原文位置
- **ExampleData**：示例数据
- **AnnotatedDocument**：带标注的完整文档

---

### 提供者层 providers/

**专业定义**：providers 目录包含各种 LLM 提供者的实现，支持 Google Gemini、OpenAI、Ollama 等，以及插件系统。

**通俗解释与类比**：providers 就像"万能电源适配器"。不同品牌的电器需要不同规格的电源，但有了适配器就可以用统一的接口。不同 LLM 服务有不同的 API，但有了 providers 模块，LangExtract 可以用统一的接口调用它们。

**在此项目中的具体体现**：

- **gemini.py**：Google Gemini 提供者
- **openai.py**：OpenAI 提供者
- **ollama.py**：Ollama 本地模型提供者
- **gemini_batch.py**：Gemini 批量处理提供者
- **builtin_registry.py**：内置提供者注册表
- **plugins.py**：插件系统，支持社区提供者

---

### 可视化层 visualization.py

**专业定义**：visualization 模块负责生成交互式的 HTML 可视化报告，展示提取结果在原文中的位置。

**通俗解释与类比**：可视化层就像"画师"或"展览设计师"。当提取完成后，用户想要"看得见"结果，画师会把原文和提取结果画成一幅直观的图画，让人们一目了然。

**在此项目中的具体体现**：

visualization 模块的核心功能：
- **HTML 生成**：生成包含所有提取结果的 HTML 文件
- **高亮显示**：在原文中高亮显示提取的内容
- **交互功能**：支持点击、悬停等交互操作
- **样式定制**：支持自定义颜色、字体等样式

---

## 架构设计原则

### 1. 模块化设计

每个模块职责单一，通过标准接口通信，便于单独测试和维护。

### 2. 可扩展性

通过 provider 插件系统和 schema 约束系统，可以方便地添加新的 LLM 支持和验证规则。

### 3. 异步处理

使用 aiohttp 实现异步并行处理，提高大规模提取的效率。

### 4. 类型安全

使用 Pydantic 进行数据验证，确保类型安全。

### 5. 向后兼容

通过 _compat 模块保持向后兼容，让老代码可以继续运行。
