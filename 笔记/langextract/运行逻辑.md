# LangExtract 运行逻辑

## 概述

LangExtract 的运行逻辑展示了从用户输入到输出结构化结果的完整流程，包括文本预处理、分块、提示词构建、LLM 推理、结果验证和后处理的各个阶段。

---

## 第一步：点出运行流程组件概念

LangExtract 的运行流程由以下核心组件构成：

- **[输入预处理](#输入预处理)**：对用户输入进行验证和格式转换，就像"接单员"接收订单并检查订单是否有效
- **[文本分块](#文本分块)**：将长文档切分成小块，就像"裁缝"把大布料剪成小布料
- **[提示词构建](#提示词构建)**：为每个文本块构建提取指令，就像"编剧"为每场戏写剧本
- **[并行推理](#并行推理)**：调用 LLM 获取提取结果，就像"外交官"去 API 市场询问答案
- **[结果验证](#结果验证)**：检查提取结果是否符合模式，就像"质检员"检查产品是否合格
- **[结果合并](#结果合并)**：合并多个块的提取结果，就像"组装工"把零件组装成成品
- **[位置映射](#位置映射)**：将提取结果映射回原文位置，就像"定位仪"标记每条结果的位置

---

## 第二步：用图构建运行流程

```
[阶段1：输入预处理]
    |
    |  用户调用 lx.extract(text, prompt, examples, ...)
    v
+---------------------------------------------------------------+
|  1. 参数验证                                                   |
|     - 检查必填参数                                             |
|     - 验证数据类型                                             |
|     - 初始化默认值                                             |
|                                                               |
|  2. 格式转换                                                   |
|     - 字符串 -> Document 对象                                 |
|     - URL -> 下载文本 -> Document 对象                         |
|     - 已有 Document 对象 -> 直接使用                           |
+---------------------------------------------------------------+
    |
    v
[阶段2：文本分块]
    |
    |  tokenizer.tokenize(document, max_char_buffer)
    v
+---------------------------------------------------------------+
|  3. 文本切分                                                   |
|     - 按句子/段落切分                                          |
|     - 保持语义连贯                                            |
|     - 相邻块有重叠                                            |
|                                                               |
|  4. 位置记录                                                  |
|     - 记录每个块在原文中的起始和结束位置                        |
|     - 为后续位置映射做准备                                    |
+---------------------------------------------------------------+
    |  返回：List[TextChunk]
    v
[阶段3：提示词构建]
    |
    |  prompting.build_prompt(chunk, prompt, examples)
    v
+---------------------------------------------------------------+
|  5. 构建任务指令                                               |
|     - 添加 prompt_description                                  |
|     - 添加格式要求（JSON/YAML）                                |
|     - 添加模式约束                                             |
|                                                               |
|  6. 注入 few-shot 示例                                         |
|     - 格式化 ExampleData                                       |
|     - 插入到提示词中                                          |
|     - 确保示例格式正确                                         |
+---------------------------------------------------------------+
    |  返回：List[Prompt]
    v
[阶段4：并行推理]
    |
    |  inference.batch_generate(prompts, model_config)
    v
+---------------------------------------------------------------+
|  7. 批量请求                                                   |
|     - 并行发送多个请求（max_workers 控制）                     |
|     - 控制请求速率                                             |
|                                                               |
|  8. API 调用                                                   |
|     - 通过 provider 调用 LLM                                   |
|     - 处理 API 密钥和认证                                      |
|     - 处理响应格式                                             |
|                                                               |
|  9. 响应解析                                                   |
|     - 解析 JSON 响应                                           |
|     - 提取结构化数据                                           |
|     - 处理错误响应                                             |
+---------------------------------------------------------------+
    |  返回：List[ExtractionResult]
    v
[阶段5：结果验证]
    |
    |  schema.validate(result, schema_constraints)
    v
+---------------------------------------------------------------+
|  10. 格式验证                                                  |
|      - 检查 JSON 格式是否正确                                  |
|      - 检查必需字段是否存在                                    |
|                                                               |
|  11. 约束验证                                                  |
|      - 验证 extraction_class 是否在允许范围内                 |
|      - 验证 attributes 格式                                    |
|      - 验证文本是否来自原文                                    |
+---------------------------------------------------------------+
    |  返回：List[ValidatedExtraction]
    v
[阶段6：结果合并]
    |
    |  merge_results(validated_results, chunks)
    v
+---------------------------------------------------------------+
|  12. 去重                                                       |
|      - 识别重复的提取                                         |
|      - 合并重复项                                              |
|                                                               |
|  13. 处理重叠                                                  |
|      - 处理边界重叠的提取                                      |
|      - 选择优先级更高的结果                                   |
|                                                               |
|  14. 位置映射                                                  |
|      - 将块内位置转换为原文位置                                |
|      - 记录 source_start 和 source_end                        |
+---------------------------------------------------------------+
    |  返回：AnnotatedDocument
    v
[阶段7：输出结果]
    |
    v
+---------------------------------------------------------------+
|  15. 返回结构化结果                                            |
|      - 返回 AnnotatedDocument 或 List[AnnotatedDocument]       |
|      - 可选：生成可视化 HTML                                   |
+---------------------------------------------------------------+
```

**流程说明**：整个流程就像一条生产线。接单员（输入预处理）接收订单，裁缝（文本分块）把大布料剪成小布料，编剧（提示词构建）为每块布料设计加工方案，外交官（并行推理）去市场采购原材料，质检员（结果验证）检查产品质量，组装工（结果合并）把零件组装成成品，最后把成品（结构化结果）交给顾客。如果顾客需要，还可以让画师（可视化）生成漂亮的展示图。

## 第三步：运行流程详细解析

### 输入预处理

**专业定义**：输入预处理阶段对用户提供的原始输入进行验证、格式转换和初始化设置。

**通俗解释与类比**：就像餐厅的"接单员"。顾客下单时，接单员需要检查菜单是否有效、食材是否齐全、特殊要求是否满足等。如果一切正常，就把订单交给厨房开始制作。

**在此项目中的具体体现**：

输入预处理完成以下工作：
1. **参数验证**：检查 `text_or_documents`、`prompt_description`、`examples` 等必填参数
2. **格式检测**：判断输入是字符串、URL 还是 Document 对象列表
3. **URL 处理**：如果是 URL，下载网页内容并转换为文本
4. **文档包装**：将文本包装成 Document 对象
5. **初始化**：创建 tokenizer、inference、schema 等模块的实例

**代码示例**：

```python
# 用户可能传入的几种输入形式
text_or_documents = "这是一段很长的文本..."  # 字符串
# 或者 URL
text_or_documents = "https://example.com/article"  # URL
# 或者 Document 对象列表
text_or_documents = [Document(text="..."), Document(text="...")]
```

---

### 文本分块

**专业定义**：文本分块阶段使用 tokenizer 将长文档切分成适合 LLM 处理的较小文本块。

**通俗解释与类比**：就像"裁缝"的工作。一块大布料没法直接塞进缝纫机，需要先根据布料的特点（句子、段落）剪成小块，分批处理，最后再拼接。

**在此项目中的具体体现**：

1. **分块策略**：
   - 使用 RegexTokenizer 按句子或段落切分
   - 块大小由 `max_char_buffer` 参数控制（默认 1000 字符）
   - 相邻块之间有重叠，保持上下文连贯

2. **位置记录**：
   - 记录每个块在原文中的起始和结束位置
   - 这对于后续将提取结果映射回原文至关重要

**代码示例**：

```python
from langextract import tokenizer as tk

tok = tk.RegexTokenizer()
chunks = tok.tokenize(document, max_char_buffer=1000)

# 每个 chunk 包含：
# - text: 块文本内容
# - start: 在原文中的起始位置
# - end: 在原文中的结束位置
```

---

### 提示词构建

**专业定义**：提示词构建阶段为每个文本块构建完整的 LLM 提示词，包括任务描述、few-shot 示例和输出格式要求。

**通俗解释与类比**：就像"编剧"为每场戏写剧本。剧本要说明这是什么场景、演员要说什么台词、最终要呈现什么效果。提示词也要说明提取任务、给几个例子、要求什么样的输出格式。

**在此项目中的具体体现**：

1. **任务指令**：
   - 添加用户的 `prompt_description`
   - 添加格式要求（JSON 或 YAML）
   - 添加模式约束（如有）

2. **示例注入**：
   - 将 `ExampleData` 格式化为 few-shot 示例
   - 确保示例的 `extraction_text` 来自示例的 `text`（verbatim）
   - 按出现顺序排列示例

**代码示例**：

```python
# 构建提示词
prompt = prompting.build_prompt(
    chunk=chunk,
    prompt_description="Extract characters and emotions.",
    examples=examples,  # List[ExampleData]
    format_type="json"
)
```

---

### 并行推理

**专业定义**：并行推理阶段使用 inference 模块调用 LLM API 获取提取结果。

**通俗解释与类比**：就像"外交官"去市场采购。厨房准备好菜单后，派外交官带着菜单去各个市场（API）采购原材料。外交官可以同时去多个市场（并行），加快采购速度。

**在此项目中的具体体现**：

1. **批量请求**：
   - 使用 `max_workers` 参数控制并行度（默认 10）
   - 使用 aiohttp 进行异步 HTTP 请求

2. **API 调用**：
   - 根据 `model_id` 选择合适的 provider
   - 处理 API 密钥认证
   - 发送请求并获取响应

3. **响应解析**：
   - 解析 JSON 格式的响应
   - 提取 `extractions` 数组
   - 处理 API 错误和异常

**代码示例**：

```python
# 并行推理
results = inference.batch_generate(
    prompts=prompts,
    model_config=model_config,
    max_workers=10
)
```

---

### 结果验证

**专业定义**：结果验证阶段使用 schema 模块验证提取结果是否符合预定义的模式约束。

**通俗解释与类比**：就像"质检员"检查产品。工厂生产的产品需要符合规格，质检员会检查产品是否合格，不合格的会标记或丢弃。

**在此项目中的具体体现**：

1. **格式验证**：
   - 检查 JSON 格式是否正确
   - 检查必需字段是否存在

2. **约束验证**：
   - 验证 `extraction_class` 是否在允许范围内（如果定义了）
   - 验证 `attributes` 的类型和格式
   - 验证 `extraction_text` 是否来自原文（可配置检查）

3. **警告生成**：
   - 生成提示词对齐警告（如示例不符合最佳实践）
   - 可配置为严格模式（抛出异常）

---

### 结果合并

**专业定义**：结果合并阶段将多个文本块的提取结果合并成最终的结构化文档。

**通俗解释与类比**：就像"组装工"把零件组装成成品。裁缝把布料剪成小块，加工后再由组装工拼接成完整的成品。

**在此项目中的具体体现**：

1. **去重**：
   - 识别跨块的重复提取
   - 保留首次出现的提取

2. **处理重叠**：
   - 处理边界重叠的提取
   - 选择更完整或更早出现的结果

3. **位置映射**：
   - 将块内相对位置转换为原文绝对位置
   - 设置 `source_start` 和 `source_end` 字段

**代码示例**：

```python
# 合并结果
final_doc = merge_results(
    validated_results=results,
    chunks=chunks,
    extraction_passes=extraction_passes  # 多轮提取时迭代处理
)
```

---

### 位置映射

**专业定义**：位置映射确保每个提取结果都能精确对应到原文中的位置。

**通俗解释与类比**：就像"定位仪"或 GPS。提取结果就像地图上的标记，定位仪确保每个标记都能精确对应到现实世界中的位置，方便后续查看和验证。

**在此项目中的具体体现**：

1. **块内位置 → 原文位置**：
   - 块的起始位置 + 块内偏移 = 原文绝对位置

2. **位置验证**：
   - 验证 `extraction_text` 与原文对应位置的内容是否一致
   - 可配置是否强制验证

3. **可视化支持**：
   - 为可视化模块提供精确的位置信息
   - 支持生成高亮显示

## 多轮提取逻辑

LangExtract 支持多轮提取（`extraction_passes > 1`）来提高召回率。

```
[第一轮提取]
    |
    v
[分析结果]
    |-> 检查是否有遗漏的重要信息
    |-> 识别可能的改进点
    |
    v
[构建下一轮提示]
    |-> 添加前一轮的提取结果
    |-> 添加"继续寻找"或"补充遗漏"的指令
    |
    v
[第二轮提取]
    |
    v
[合并多轮结果]
```

每轮提取都会以上一轮的结果作为上下文，这样可以：
- 发现遗漏的信息
- 补充额外的属性
- 修正错误
