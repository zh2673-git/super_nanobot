# 核心算法分析

> 本笔记深入分析 aimirror 的核心算法原理，帮助零基础用户理解关键技术。

## 一、算法概述

### 第一步：点出核心算法

aimirror 包含5个核心算法：

- **HTTP Range 分片下载算法**：将大文件分成多个小块并行下载
- **LRU 缓存淘汰算法**：自动清理最久未使用的缓存文件
- **SHA256 文件去重算法**：使用哈希值避免重复存储
- **并发控制算法**：限制同时下载的任务数，防止资源耗尽
- **长轮询等待算法**：优化客户端等待正在下载文件的体验

## 二、HTTP Range 分片下载算法

### 第一步：点出算法概念

HTTP Range 分片下载算法利用HTTP协议的Range请求头，将大文件分成多个小块并行下载。

- **Range请求头**：告诉服务器只返回文件的指定部分，如 `Range: bytes=0-1023`
- **206 Partial Content**：服务器返回部分内容的状态码
- **并发下载**：多个线程同时下载不同分片，充分利用带宽

### 第二步：用图展示算法流程

```
[输入: URL, 文件大小, 并发数, 分片大小]
    │
    ▼
[计算分片数量]
    │
    │  分片数量 = ceil(文件大小 / 分片大小)
    │
    ▼
[生成分片列表]
    │
    │  分片1: start=0, end=chunk_size-1
    │  分片2: start=chunk_size, end=2*chunk_size-1
    │  ...
    │  分片N: start=(N-1)*chunk_size, end=file_size-1
    │
    ▼
[并发下载分片]
    │
    ├─ 线程1: Range: bytes=0-1048575 ────────┐
    ├─ 线程2: Range: bytes=1048576-2097151 ──┤
    ├─ 线程3: Range: bytes=2097152-3145727 ──┼─ 并行执行
    ├─ ...                                   │
    └─ 线程20: Range: bytes=...             ┘
    │
    ▼
[等待所有分片下载完成]
    │
    ▼
[按序写入文件]
    │
    │  seek(start_position)
    │  write(chunk_data)
    │
    ▼
[验证文件完整性]
    │
    │  检查文件大小是否正确
    │
    ▼
[输出: 下载完成的文件]
```

### 第三步：算法详细解析

#### 1. 分片计算

**专业定义**：根据文件总大小和分片大小，计算每个分片的起始和结束位置。

**通俗解释与类比**：就像把一根长绳子切成多段，每段长度相等（最后一段可能短一些），然后分配给不同的人同时搬运。

**代码实现**：
```python
def _split_chunks(self, total_size: int) -> List[Chunk]:
    """分割下载范围"""
    # 自动模式：当 chunk_size <= 0 时，自动计算
    if self.chunk_size <= 0:
        # 扣除10个线程预留给其他任务
        chunk_size = total_size // max((self.concurrency - 10), 1)
        if chunk_size <= 0:
            chunk_size = total_size
    else:
        chunk_size = self.chunk_size
    
    chunks = []
    for start in range(0, total_size, chunk_size):
        end = min(start + chunk_size - 1, total_size - 1)
        chunks.append(Chunk(start=start, end=end))
    return chunks
```

**在此算法中的具体体现**：
- **自动模式**：`chunk_size = total_size / (concurrency - 10)`
  - 为什么减10？预留给其他下载任务，避免所有线程被一个任务占用
- **最后分片**：`end = total_size - 1`，确保不超出文件末尾
- **示例**：10GB文件，20并发，自动模式
  - `chunk_size = 10GB / 10 = 1GB`
  - 分片数 = 10个，每片1GB

#### 2. 并发下载

**专业定义**：使用异步IO（aiohttp）并发发送多个HTTP Range请求，每个请求只下载文件的一部分。

**通俗解释与类比**：就像20个人同时搬运一堆砖头，每个人搬运一部分，最后一起堆好。比一个人搬完所有砖头快很多。

**代码实现**：
```python
async def _download_chunk(self, session, chunk, sem, retry=3):
    """下载单个分片，带重试机制"""
    async with sem:  # 控制并发数
        headers = {'Range': f'bytes={chunk.start}-{chunk.end}'}
        
        for attempt in range(retry):
            try:
                async with session.get(self.url, headers=headers) as resp:
                    if resp.status == 206:  # Partial Content
                        chunk.data = await resp.read()
                        return
                    else:
                        raise RuntimeError(f"Failed: {resp.status}")
            except Exception as e:
                if attempt < retry - 1:
                    await asyncio.sleep(2 ** attempt)  # 指数退避
                else:
                    raise
```

**在此算法中的具体体现**：
- **Semaphore**：控制同时下载的分片数不超过 `concurrency`
- **重试机制**：失败后等待 `2^attempt` 秒再重试（指数退避）
- **206状态码**：服务器正确返回部分内容

#### 3. 文件写入

**专业定义**：将下载的分片数据按序写入文件的对应位置，确保数据完整性。

**通俗解释与类比**：就像拼图，每块拼图都有固定的位置，必须放在正确的地方才能组成完整画面。

**代码实现**：
```python
async with aiofiles.open(temp_file, 'r+b') as f:
    for chunk in sorted(self.chunks, key=lambda c: c.start):
        await f.seek(chunk.start)  # 定位到分片起始位置
        await f.write(chunk.data)  # 写入数据
```

**在此算法中的具体体现**：
- **预分配空间**：`truncate(total_size)` 创建指定大小的空文件
- **按序写入**：必须按 `start` 排序，避免数据错位
- **随机写入**：`seek()` 定位到任意位置写入

### 性能分析

**时间复杂度**：
- 下载时间：`O(file_size / (bandwidth * concurrency))`
- 写入时间：`O(file_size / disk_speed)`
- 总时间：`max(下载时间, 写入时间)`

**空间复杂度**：
- 内存：`O(chunk_size * concurrency)`（存储正在下载的分片）
- 磁盘：`O(file_size)`（存储完整文件）

**优化效果**：
- 单线程：瓶颈在网络带宽，`时间 = 文件大小 / 带宽`
- 并行下载：`时间 = 文件大小 / (带宽 * 并发数)`

## 三、LRU 缓存淘汰算法

### 第一步：点出算法概念

LRU（Least Recently Used，最近最少使用）是一种缓存淘汰策略，当缓存满时，删除最久未被访问的文件。

- **访问时间**：每次访问文件时更新时间戳
- **淘汰策略**：删除最久未访问的文件
- **SQLite索引**：使用数据库索引加速访问时间查询

### 第二步：用图展示算法流程

```
[存入缓存: cache.put(url, filepath)]
    │
    ▼
[检查缓存容量]
    │
    ├─ 总大小 <= 限制: 结束
    │
    └─ 总大小 > 限制: 执行淘汰
         │
         ▼
    [查找最久未访问的文件]
         │
         │  SELECT * FROM files ORDER BY accessed ASC LIMIT 1
         │
         ▼
    [删除文件]
         │
         │  os.remove(filepath)
         │  DELETE FROM files WHERE digest=?
         │
         ▼
    [重复直到满足容量]
         │
         └─ 返回步骤"检查缓存容量"
```

### 第三步：算法详细解析

#### 1. 访问时间更新

**专业定义**：每次访问缓存文件时，更新其访问时间戳。

**通俗解释与类比**：就像超市的商品，每次被拿起来看，就记一次时间。清理时，把很久没人碰的商品扔掉。

**代码实现**：
```python
def get(self, url: str, content_type: str = "") -> Optional[str]:
    digest = self._get_digest(url, content_type)
    conn = sqlite3.connect(self.db_path)
    c = conn.cursor()
    c.execute('SELECT filepath, size FROM files WHERE digest=?', (digest,))
    row = c.fetchone()
    conn.close()
    
    if row and os.path.exists(row[0]):
        # 更新访问时间
        conn = sqlite3.connect(self.db_path)
        conn.execute('UPDATE files SET accessed=CURRENT_TIMESTAMP WHERE digest=?', (digest,))
        conn.commit()
        conn.close()
        return row[0]
    return None
```

**在此算法中的具体体现**：
- **CURRENT_TIMESTAMP**：SQLite函数，返回当前时间
- **UPDATE语句**：更新 `accessed` 字段为当前时间

#### 2. 淘汰策略

**专业定义**：当缓存超过容量限制时，找出访问时间最早的文件并删除。

**通俗解释与类比**：就像清理冰箱，先拿出最早过期的食物扔掉，直到冰箱有空间放新食物。

**代码实现**：
```python
def _evict_if_needed(self):
    """LRU 淘汰策略"""
    conn = sqlite3.connect(self.db_path)
    c = conn.cursor()
    
    # 计算当前总大小
    c.execute('SELECT SUM(size) FROM files')
    total = c.fetchone()[0] or 0
    
    while total > self.max_bytes:
        # 找出最久未访问的文件
        c.execute('''
            SELECT digest, filepath, size FROM files 
            ORDER BY accessed ASC LIMIT 1
        ''')
        row = c.fetchone()
        if not row:
            break
        digest, filepath, size = row
        
        # 删除文件
        if os.path.exists(filepath):
            os.remove(filepath)
        c.execute('DELETE FROM files WHERE digest=?', (digest,))
        total -= size
    
    conn.commit()
    conn.close()
```

**在此算法中的具体体现**：
- **ORDER BY accessed ASC**：按访问时间升序排序
- **LIMIT 1**：只取最早的一个
- **循环淘汰**：可能需要删除多个文件才能满足容量
- **事务保证**：使用SQLite事务保证数据一致性

#### 3. 索引优化

**专业定义**：在 `accessed` 字段上创建索引，加速排序查询。

**通俗解释与类比**：就像图书馆按借阅时间给书排序，如果每次清理都要翻遍所有书，太慢了。建立一个索引，快速找到最早借阅的书。

**代码实现**：
```python
c.execute('CREATE INDEX IF NOT EXISTS idx_accessed ON files(accessed)')
```

**在此算法中的具体体现**：
- **索引**：SQLite的B-Tree索引，加速 `ORDER BY accessed ASC` 查询
- **查询复杂度**：从 `O(n)` 降低到 `O(log n)`

### 性能分析

**时间复杂度**：
- 访问更新：`O(1)`（直接更新时间戳）
- 淘汰查找：`O(log n)`（索引查询）
- 删除：`O(m)`（m为需要删除的文件数）

**空间复杂度**：
- 数据库：`O(n)`（n为缓存文件数）
- 索引：`O(n)`（额外空间）

**优化效果**：
- 无索引：查找最久未访问的文件需要 `O(n)`
- 有索引：查找最久未访问的文件只需 `O(log n)`

## 四、SHA256 文件去重算法

### 第一步：点出算法概念

SHA256是一种密码学哈希算法，将任意长度的输入映射为固定长度（256位）的输出。

- **哈希值**：URL的固定长度指纹，如 `sha256(url) = "abc123..."`
- **去重**：相同URL生成相同哈希值，避免重复存储
- **硬链接**：多个URL映射到同一物理文件

### 第二步：用图展示算法流程

```
[输入: URL]
    │
    ▼
[计算SHA256哈希值]
    │
    │  digest = hashlib.sha256(url.encode()).hexdigest()
    │
    ▼
[检查缓存]
    │
    ├─ 存在: 返回缓存文件路径
    │
    └─ 不存在: 下载并存入缓存
         │
         ▼
    [下载文件到临时路径]
         │
         ▼
    [创建硬链接或复制]
         │
         │  硬链接: os.link(filepath, cache_dir/digest)
         │  复制: shutil.copy2(filepath, cache_dir/digest)
         │
         ▼
    [更新SQLite数据库]
         │
         │  INSERT INTO files (digest, filepath, size, ...)
         │
         ▼
    [返回缓存文件路径]
```

### 第三步：算法详细解析

#### 1. 哈希计算

**专业定义**：使用SHA256算法计算URL的哈希值，作为缓存键。

**通俗解释与类比**：就像给每个包裹贴上唯一的标签，标签上有64个字符的编号，不同包裹编号不同，相同包裹编号相同。

**代码实现**：
```python
def _get_digest(self, url: str, content_type: str = "") -> str:
    """生成缓存键：url 的 sha256"""
    return hashlib.sha256(url.encode()).hexdigest()
```

**在此算法中的具体体现**：
- **SHA256**：密码学哈希算法，碰撞概率极低
- **hexdigest()**：返回16进制字符串，共64个字符
- **URL作为输入**：相同的URL生成相同的哈希值

#### 2. 硬链接

**专业定义**：创建指向同一文件的硬链接，不占用额外磁盘空间。

**通俗解释与类比**：就像给同一本书起了多个名字，放在不同的书架上，但实际上只有一本书。读任何一个名字的书，内容都一样。

**代码实现**：
```python
target = self.cache_dir / digest
if not os.path.exists(target):
    try:
        os.link(filepath, target)  # 硬链接
    except OSError:
        # 跨文件系统时硬链接失败，使用复制
        shutil.copy2(filepath, target)
```

**在此算法中的具体体现**：
- **os.link()**：创建硬链接，指向同一inode
- **空间节省**：硬链接不占用额外磁盘空间
- **跨文件系统**：无法创建硬链接时使用复制

### 性能分析

**时间复杂度**：
- 哈希计算：`O(n)`（n为URL长度，通常很小）
- 查询：`O(1)`（SQLite主键查询）

**空间复杂度**：
- 数据库：`O(1)`（固定大小的哈希值）
- 硬链接：`O(0)`（不占用额外空间）
- 复制：`O(file_size)`（复制整个文件）

## 五、并发控制算法

### 第一步：点出算法概念

并发控制算法使用信号量（Semaphore）限制同时进行的下载任务数，防止资源耗尽。

- **全局并发限制**：整个服务最多同时进行的下载任务数
- **信号量**：计数器，获取时减1，释放时加1，为0时等待
- **公平性**：先到先得，避免饥饿

### 第二步：用图展示算法流程

```
[新下载请求]
    │
    ▼
[尝试获取信号量]
    │
    ├─ 信号量 > 0: 获取成功，继续
    │    │
    │    └─ 信号量 -= 1
    │
    └─ 信号量 = 0: 等待
         │
         └─ 阻塞直到其他任务释放信号量
              │
              └─ 唤醒后继续
    │
    ▼
[执行下载任务]
    │
    ▼
[释放信号量]
    │
    └─ 信号量 += 1
```

### 第三步：算法详细解析

#### 1. 信号量初始化

**专业定义**：创建一个计数信号量，初始值为最大并发数。

**通俗解释与类比**：就像停车场有100个车位，每个车位是一个信号量。车来时占用一个车位（减1），车走时空出一个车位（加1）。如果车位满了，新车必须等有车离开才能进入。

**代码实现**：
```python
max_concurrent_downloads = config.get('server', {}).get('max_concurrent_downloads', 10)
download_semaphore = asyncio.Semaphore(max_concurrent_downloads)
```

**在此算法中的具体体现**：
- **默认值**：10个并发任务
- **配置化**：可通过 `config.yaml` 调整
- **asyncio.Semaphore**：异步信号量，支持await

#### 2. 信号量获取与释放

**专业定义**：使用 `async with` 自动获取和释放信号量。

**通俗解释与类比**：就像借书，用 `async with` 自动办理借书手续（获取），还书时自动办理归还手续（释放）。

**代码实现**：
```python
async with download_semaphore:
    # 获取信号量成功，开始下载
    ...
    # 下载完成，自动释放信号量
```

**在此算法中的具体体现**：
- **async with**：确保即使下载失败也能释放信号量
- **阻塞等待**：信号量为0时，await会阻塞
- **公平调度**：FIFO调度，先到先得

### 性能分析

**时间复杂度**：
- 获取信号量：`O(1)`（可用时）
- 等待信号量：`O(n)`（n为等待的任务数）

**空间复杂度**：
- 信号量：`O(1)`（只有一个计数器）

**并发效果**：
- 无限制：可能同时启动数千个任务，资源耗尽
- 有限制：最多 `max_concurrent_downloads` 个任务同时运行

## 六、长轮询等待算法

### 第一步：点出算法概念

长轮询等待算法用于处理多个客户端同时请求同一文件的情况，避免重复下载。

- **正在下载列表**：记录正在下载的文件
- **轮询检查**：定时检查下载是否完成
- **超时重试**：超过时间返回302让客户端重试

### 第二步：用图展示算法流程

```
[新请求到达]
    │
    ▼
[检查缓存]
    │
    ├─ 命中: 返回缓存文件
    │
    └─ 未命中: 检查是否正在下载
         │
         ├─ 正在下载: 长轮询等待
         │    │
         │    ├─ 每0.5秒检查一次
         │    │    │
         │    │    ├─ 下载完成: 返回缓存文件
         │    │    │
         │    │    └─ 未完成: 继续等待
         │    │
         │    ├─ 超时(20秒): 返回302重试
         │    │
         │    └─ 客户端断开: 后台下载继续
         │
         └─ 未在下载: 开始新下载
              │
              └─ 标记为正在下载
```

### 第三步：算法详细解析

#### 1. 正在下载检查

**专业定义**：维护一个字典，记录正在下载的文件及其任务。

**通俗解释与类比**：就像图书馆的书借出记录，如果有人正在借某一本书，其他人想借就要等。

**代码实现**：
```python
# 全局变量
active_downloads: dict = {}  # {cache_key: (temp_file, task, start_time)}

# 检查是否正在下载
if cache_key in active_downloads:
    temp_file, download_task, start_time = active_downloads[cache_key]
    # 长轮询等待
```

**在此算法中的具体体现**：
- **字典键**：缓存键（URL的哈希值）
- **字典值**：临时文件路径、下载任务、开始时间
- **任务完成检查**：`download_task.done()`

#### 2. 长轮询实现

**专业定义**：客户端定期检查下载状态，直到完成或超时。

**通俗解释与类比**：就像等公交车，每分钟看一次车来了没有，等20分钟还没来就回家明天再来。

**代码实现**：
```python
# 长轮询：每0.5秒检查一次，最多20秒
max_wait = 20
poll_interval = 0.5
poll_count = int(max_wait / poll_interval)

for _ in range(poll_count):
    await asyncio.sleep(poll_interval)
    
    if download_task.done():
        exc = download_task.exception()
        if exc:
            raise HTTPException(502, f"Download failed: {str(exc)}")
        
        # 等待缓存写入
        for _ in range(10):
            cached = cache.get(cache_key, content_type)
            if cached:
                return StreamingResponse(...)
            await asyncio.sleep(0.5)
        break

# 超时：返回302重试
return Response(status_code=302, headers={"Location": str(request.url)})
```

**在此算法中的具体体现**：
- **轮询次数**：`20秒 / 0.5秒 = 40次`
- **任务完成检查**：`download_task.done()`
- **异常检查**：`download_task.exception()`
- **缓存写入等待**：最多等待5秒
- **超时处理**：返回302让客户端重试

#### 3. 后台下载继续

**专业定义**：即使客户端断开连接，后台下载任务继续运行。

**通俗解释与类比**：就像外卖店，即使顾客取消订单，厨房里的菜继续做，做完存起来给下一个顾客。

**代码实现**：
```python
except asyncio.CancelledError:
    # 客户端断开连接，后台下载继续运行
    logging.info(f"Client disconnected, download continuing in background")
    raise  # 重新抛出，让 FastAPI 处理断开连接
```

**在此算法中的具体体现**：
- **CancelledError**：客户端断开连接时抛出
- **后台任务**：下载任务在后台继续运行
- **缓存存储**：下载完成后存入缓存供下次使用

### 性能分析

**时间复杂度**：
- 轮询检查：`O(1)`（每次检查）
- 总等待时间：最多20秒

**空间复杂度**：
- 字典：`O(n)`（n为正在下载的任务数）

**优化效果**：
- 无长轮询：客户端频繁重试，增加服务器压力
- 有长轮询：客户端等待，减少重复请求

## 七、算法对比表

| 算法 | 时间复杂度 | 空间复杂度 | 优化目标 |
|------|-----------|-----------|----------|
| HTTP Range分片下载 | `O(file_size / (bandwidth * concurrency))` | `O(chunk_size * concurrency)` | 提高下载速度 |
| LRU缓存淘汰 | `O(log n)` 查找 + `O(m)` 删除 | `O(n)` 数据库 | 限制缓存容量 |
| SHA256去重 | `O(url_length)` | `O(1)` 哈希值 | 避免重复存储 |
| 并发控制 | `O(1)` 获取 | `O(1)` 信号量 | 限制资源使用 |
| 长轮询等待 | `O(1)` 检查 | `O(n)` 字典 | 优化客户端体验 |
