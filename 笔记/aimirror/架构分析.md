# 架构分析

> 本笔记深入解析 aimirror 项目的架构设计，帮助零基础用户理解模块关系和设计理念。

## 一、架构概述

### 第一步：点出架构的核心概念

理解 aimirror 架构需要认识5个核心概念：

- **路由层**：像"导航员"，根据URL特征选择最佳下载策略和上游服务器
- **处理器层**：像"专业翻译"，处理Docker等复杂协议的特殊认证和重定向逻辑
- **下载层**：像"物流中心"，负责任务调度、分片下载和并发控制
- **缓存层**：像"仓库"，存储已下载文件并提供快速检索
- **配置层**：像"说明书"，定义规则、策略和参数

## 三、核心设计模式

### 1. 策略模式（Strategy Pattern）

**专业解释**：aimirror使用策略模式来实现不同的下载策略。根据路由规则，可以选择`proxy`策略（直接代理）或`parallel`策略（并行下载）。

**通俗解释与类比**：就像送快递有不同的方式：
- **proxy策略**：像普通快递，一个快递员送一个包裹，简单直接
- **parallel策略**：像分段快递，多个快递员同时送一个大包裹的不同部分，最后合并

**在架构中的体现**：
```yaml
# config.yaml 中的策略配置
rules:
  - name: docker-blob
    strategy: parallel    # 使用并行策略
    concurrency: 20       # 20个并发
    
  - name: docker-registry
    strategy: proxy       # 使用代理策略
    handler: handlers.docker
```

### 2. 责任链模式（Chain of Responsibility）

**专业解释**：请求处理流程采用责任链模式，每个处理节点可以中断或继续传递请求。

**通俗解释与类比**：就像医院的分诊流程：
1. 先挂号的分诊台（main.py）
2. 再到科室判断（router.py）
3. 如果是Docker，转到专科诊室（handlers/docker.py）
4. 最后治疗或检查（downloader.py / cache.py）

**在架构中的体现**：
```
请求 -> main.py -> router.py -> handlers -> downloader/cache -> 响应
        ↓           ↓             ↓              ↓
      判断中断    判断中断      判断中断        最终处理
```

### 3. 观察者模式（Observer Pattern）

**专业解释**：缓存系统采用LRU（最近最少使用）淘汰策略，类似观察者模式中的事件通知机制。

**通俗解释与类比**：就像仓库管理：
- 每次取货（访问缓存），记录时间戳
- 当仓库快满时，清理最久没动的货物（LRU淘汰）

**在架构中的体现**：
```python
# cache.py 中的LRU淘汰
def _evict_if_needed(self):
    # 计算总大小
    # 找出最久未访问的文件
    # 删除文件并更新数据库
```

## 四、模块依赖关系

### 依赖图

```
main.py
  │
  ├── router.py (路由匹配)
  ├── cache.py (缓存管理)
  ├── downloader.py (并行下载)
  └── handlers/docker.py (Docker特殊处理)
        │
        └── 依赖: main.py的全局变量（config, http_client）

外部依赖:
  ├── fastapi (Web框架)
  ├── httpx (HTTP客户端)
  ├── aiohttp (异步HTTP客户端)
  ├── aiofiles (异步文件操作)
  └── sqlite3 (元数据存储)
```

### 模块职责

| 模块 | 依赖模块 | 职责 |
|------|---------|------|
| `main.py` | router, cache, downloader, handlers | 服务生命周期、请求分发 |
| `router.py` | 无 | URL规则匹配、策略选择 |
| `cache.py` | 无 | 缓存存取、LRU淘汰、SQLite管理 |
| `downloader.py` | 无 | 文件大小获取、分片计算、并发下载 |
| `handlers/docker.py` | main.py (全局变量) | Docker认证、重定向、token代理 |

## 五、数据流分析

### 请求处理流程

```
[客户端请求]
    │
    ▼
[健康检查 /stats /health] ────► 直接返回统计信息
    │
    ▼
[路由匹配 router.match(path)]
    │
    ├─── 匹配失败 ────► 返回 404
    │
    ├─── 策略: proxy ────► _proxy_request()
    │                        │
    │                        ├─── 缓存命中 ────► 返回缓存文件流
    │                        │
    │                        └─── 代理转发 ────► 返回上游响应
    │
    └─── 策略: parallel ────► _parallel_download()
                               │
                               ├─── Handler处理 ────► 特殊逻辑处理
                               │
                               ├─── HEAD获取元数据
                               │
                               ├─── 缓存命中 ────► 返回缓存文件流
                               │
                               ├─── 正在下载 ────► 长轮询等待
                               │
                               └─── 开始下载
                                     │
                                     ├─── 并发分片下载
                                     ├─── 实时写入文件
                                     ├─── 存入缓存
                                     └─── 返回文件流
```

### 并发控制机制

```
全局并发限制（download_semaphore）
    │
    ├─── 最大并发数: 100（可配置）
    │
    ├─── 获取信号量 ────► 开始下载
    │
    └─── 释放信号量 ────► 下载完成（后台任务继续）

单个下载任务的并发控制
    │
    ├─── 分片数: chunk_size 自动计算或固定值
    │
    ├─── 下载数: concurrency (如20)
    │
    └─── Semaphore(concurrency) 控制并发分片数
```

## 六、配置系统设计

### 配置层级

```yaml
config.yaml
├─ server（服务配置）
│   ├─ host/port: 监听地址
│   ├─ upstream_proxy: 上游代理
│   ├─ public_host: 对外访问地址
│   └─ max_concurrent_downloads: 全局并发限制
│
├─ cache（缓存配置）
│   ├─ dir: 缓存目录
│   ├─ max_size_gb: 最大容量
│   └─ lru_enabled: 启用LRU淘汰
│
├─ rules（路由规则）
│   ├─ name: 规则名称
│   ├─ pattern: URL正则
│   ├─ upstream: 上游地址
│   ├─ strategy: 策略(proxy/parallel)
│   ├─ handler: 处理器路径（可选）
│   └─ 其他参数（min_size, concurrency等）
│
└─ logging（日志配置）
    ├─ level: 日志级别
    └─ file: 日志文件路径
```

### 规则优先级

```
路由匹配顺序:
    规则1 -> 规则2 -> 规则3 -> ... -> default规则
    
匹配规则:
    - 使用正则表达式匹配URL路径
    - 返回第一个匹配的规则
    - default规则作为兜底（必须存在）
```

## 七、扩展点设计

### 1. Handler扩展机制

**如何添加新的Handler**：

```python
# handlers/custom.py

async def exec_path(request, full_path, config, http_client):
    """
    自定义处理器
    
    Args:
        request: FastAPI Request对象
        full_path: 请求路径
        config: 全局配置字典
        http_client: httpx.AsyncClient实例
    
    Returns:
        (handled, response): 
        - handled: bool，是否已处理
        - response: Response对象或None
    """
    if full_path.startswith('/custom/'):
        # 处理自定义逻辑
        return True, Response(content="Custom response")
    
    # 未处理，继续后续流程
    return False, None
```

**配置**：
```yaml
rules:
  - name: custom-handler
    pattern: "/custom/.*"
    upstream: "https://example.com"
    strategy: proxy
    handler: handlers.custom
```

### 2. 策略扩展机制

**新增下载策略**：

```python
# 在 main.py 中添加新的策略处理函数
async def _new_strategy_download(request: Request, target_url: str, rule) -> Response:
    # 自定义下载逻辑
    pass

# 在 proxy_handler 中添加分支
if rule.strategy == 'new_strategy':
    return await _new_strategy_download(request, target_url, rule)
```

### 3. 缓存后端切换

**当前设计**：使用文件系统 + SQLite

**可扩展点**：
- 替换为Redis缓存
- 替换为S3存储
- 替换为分布式缓存

## 八、性能优化设计

### 1. 连接池优化

```python
# main.py 中的连接池配置
limits = httpx.Limits(
    max_connections=200,           # 最大连接数
    max_keepalive_connections=50,  # 最大保持连接数
    keepalive_expiry=30.0          # 保持连接过期时间
)
```

**作用**：避免频繁建立/关闭连接，提高吞吐量。

### 2. 流式返回设计

```python
# 边下载边返回数据
async def _streaming_file_iterator(filepath, download_task, total_size):
    while last_size < total_size:
        # 等待数据准备好
        # 立即返回已准备好的数据
        yield chunk
```

**作用**：客户端不需要等待整个文件下载完成，可以边下载边接收数据。

### 3. 长轮询机制

```python
# 正在下载的文件，客户端可以等待
for _ in range(40):  # 最多等待20秒
    await asyncio.sleep(0.5)
    if download_task.done():
        return cached_file
# 超时返回302让客户端重试
return Response(status_code=302, headers={"Location": url})
```

**作用**：避免客户端频繁重试，减轻服务器压力。
